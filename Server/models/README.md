# models/

**Словарь данных всего проекта.** Здесь лежат структуры, которыми оперируют
все остальные слои: что такое «пользователь», «сессия», «личная комната».
Прежде чем читать `services/` или `network/`, прочитай эту папку — там термины,
на которых написано остальное.

Для того, кто пишет **клиент** (Qt-десктоп, Android), это ещё и **контракт
обмена**: методы `*Json()` внизу каждой модели показывают точную структуру
JSON, которую сервер присылает и которую клиент должен разобрать. Скопировав
эти поля в свой парсер, ты получишь совместимость с сервером, не читая
сетевой код.

## Что такое «модель» в этом проекте

Модель — это простой `struct` **без поведения**: набор полей + пара чистых
функций-сериализаторов. Никакой логики, никаких обращений к базе или сети.
Приём взят из [MedFlow](https://github.com/vi5vi5vi5/medflow).

Три правила, общие для всех моделей:

1. **`id = -1` означает «ещё не сохранён».** Новый объект создаётся в памяти с
   `id == -1`; при первом `save()` репозиторий (см. [`../interface`](../interface/README.md))
   присваивает настоящий `id` из базы. Так по значению `id` всегда видно,
   новая это запись или уже существующая.
2. **Поля-`*Ms`** (`createdAtMs`, `expiresAtMs`, …) — это Unix-время в
   **миллисекундах** (`qint64`). Не секунды, не `QDateTime`. Клиент получает
   их числом и сам решает, как показать.
3. **Модель ничего не проверяет.** Валидация логина, длины имени, силы пароля —
   это [`../services`](../services/README.md). Модель лишь хранит уже готовое
   значение. Комментарии вида «нормализован: lowercase, 3–32» описывают
   *инвариант, который гарантирует сервис*, а не проверку внутри модели.

## Модели

| Файл | Модель(и) | Одной фразой | Где хранится |
|---|---|---|---|
| `User.h` | `User` | Аккаунт: логин, имя, производные пароля, аватар | таблица `users` |
| `Session.h` | `Session` | Активный вход: токен в куке → пользователь | таблица `sessions` |
| `PersonalRoom.h` | `PersonalRoom` | Личная комната: постоянный код-ссылка + пароль | таблица `personal_rooms` |
| `RoomAlias.h` | `RoomAlias` | Доп. ссылка на комнату со своими правилами входа | таблица `room_aliases` |

> Один файл — одна модель. `ChatEntry` (сообщение чата) сюда **не входит**: он
> живёт только в памяти комнаты и объявлен в [`../core/ConferenceRoom.h`](../core/README.md),
> а не здесь, потому что в базу не сохраняется.

### `User` — аккаунт

Центральная модель. На неё ссылаются все остальные (`Session.userId`,
`PersonalRoom.ownerId`).

| Поле | Тип | Смысл |
|---|---|---|
| `id` | `int` | Первичный ключ. `-1` — не сохранён. |
| `login` | `QString` | Уникальный, нормализованный: lowercase, 3–32 символа, `[a-z0-9_.-]`. Гарантирует `AuthService::normalizeLogin`. |
| `displayName` | `QString` | Отображаемое имя (1–40 символов). Его видят участники конференции; логин — нет. |
| `passAlgo` | `QString` | Алгоритм хеша (`"pbkdf2-sha512"`). Заложено на будущую миграцию алгоритма. |
| `passIters` | `int` | Число итераций PBKDF2 для этого пользователя. Хранится в строке, чтобы стоимость можно было поднять, не ломая старые хеши. |
| `passSalt` | `QByteArray` | 16 случайных байт соли. |
| `passHash` | `QByteArray` | 64 байта `PBKDF2-HMAC-SHA512(password, salt, iters)`. |
| `avatarVer` | `int` | Версия аватарки. `0` — аватарки нет. Растёт на 1 при каждой загрузке — используется для сброса кэша (URL `?v=<avatarVer>`). |
| `createdAtMs` | `qint64` | Когда создан аккаунт. |

**Про пароль — важно.** Сам пароль **нигде не хранится** и восстановлению не
подлежит. В базе только соль + хеш + число итераций. Проверка пароля — это
пере-вычисление хеша и сравнение (см. [`AuthService`](../services/README.md)).
Поля `pass*` наружу **никогда** не отдаются.

**JSON для клиента — `publicJson()`:**

```jsonc
{
  "id": 42,
  "login": "vi5",
  "display_name": "Пётр",
  "avatar_ver": 3
}
```

Это **единственное** представление `User` для клиента. Обрати внимание: полей
`pass*` и `createdAtMs` здесь нет — сериализатор намеренно отдаёт только
безопасное подмножество. Клиент получает этот объект в ответах
`/api/auth/register`, `/api/auth/login`, `/api/me` (внутри `{"user": …}`).

Аватарку в JSON не кладут — она отдаётся отдельным эндпоинтом
`/api/users/<id>/avatar?v=<avatar_ver>` как JPEG. Клиент строит этот URL сам
из `id` и `avatar_ver`.

### `Session` — активный вход

| Поле | Тип | Смысл |
|---|---|---|
| `token` | `QString` | 32 случайных байта в base64url. **Он же первичный ключ** — отдельный `id` не нужен. Живёт в HttpOnly-куке `meetup_session`. |
| `userId` | `int` | Какому `User::id` принадлежит сессия. |
| `createdAtMs` | `qint64` | Когда вошли. |
| `expiresAtMs` | `qint64` | Когда сессия протухнет. Продлевается на лету (скользящее окно) в `AuthService`. |

**Метод:** `isExpired(nowMs)` — `true`, если `nowMs >= expiresAtMs`. Всё, чем
модель «умеет» пользоваться сама: чистое сравнение без побочных эффектов.

**JSON нет — и это осознанно.** `Session` целиком серверная: токен уходит к
клиенту только внутри заголовка `Set-Cookie`, и то как строка. Клиент токен
**не видит** (кука HttpOnly) и не хранит — браузер шлёт её сам. Для нативного
клиента (Qt/Android) это значит: после входа нужно сохранить cookie из ответа
и прикладывать её к последующим запросам; содержимое токена парсить не надо.

### `PersonalRoom` — личная комната

Постоянная «моя» конференция пользователя: у неё запоминающийся код-ссылка
(`?room=vi5`), название и необязательный пароль. **Одна на пользователя**
(в базе `owner_id UNIQUE`).

| Поле | Тип | Смысл |
|---|---|---|
| `id` | `int` | Первичный ключ. `-1` — не сохранена. |
| `ownerId` | `int` | `User::id` владельца. Уникален: у пользователя не больше одной комнаты. |
| `code` | `QString` | Постоянный код входа. Нормализован: lowercase, 3–32, `[a-z0-9_-]`. Уникален среди всех комнат и алиасов. |
| `title` | `QString` | Название (1–64 символа). Видят участники. |
| `password` | `QString` | Пароль на вход. Пусто — вход свободный. |
| `createdAtMs` | `qint64` | Когда создана. |

**Почему пароль открытым текстом.** В отличие от пароля аккаунта, это пароль
*комнаты*: владелец должен уметь «подсмотреть» его в настройках, чтобы
продиктовать гостю. Поэтому он хранится как есть (согласовано в дорожной
карте). Аккаунтные пароли по-прежнему только PBKDF2 — не путать.

**JSON для владельца — `ownerJson()`:**

```jsonc
{
  "code": "vi5",
  "title": "Личная переговорка",
  "password": "secret"   // владелец вправе видеть свой пароль
}
```

Отдаётся владельцу в `/api/me/room`. Прочим участникам пароль, разумеется, не
показывают — `network/HttpApi` для чужого взгляда собирает урезанный объект
вручную (только `title` + флаг `has_password`), не вызывая `ownerJson()`.

### `RoomAlias` — доп. ссылка со своими правилами

Дополнительный код входа в ту же `PersonalRoom`, но с **отдельными** правилами:
свой пароль, лимит использований, белый список логинов, выключатель. Позволяет
раздать разным людям разные одноразовые/ограниченные ссылки на одну комнату.
Максимум 5 на комнату; код генерирует сервер (случайный).

| Поле | Тип | Смысл |
|---|---|---|
| `id` | `int` | Первичный ключ. `-1` — не сохранён. |
| `roomId` | `int` | В какую `PersonalRoom::id` ведёт. |
| `code` | `QString` | Случайный код, уникальный среди алиасов и комнат. |
| `password` | `QString` | Свой пароль на вход. Пусто — без пароля. |
| `usesLeft` | `int` | Сколько входов осталось. `-1` — без лимита. Дойдя до `0`, алиас удаляется — так делается «одноразовая ссылка». |
| `logins` | `QStringList` | Белый список логинов. Пусто — пускать всех; иначе только перечисленных (анонимов список отсекает всегда). |
| `enabled` | `bool` | Выключенный алиас неотличим от несуществующего. |
| `createdAtMs` | `qint64` | Когда создан. |

**JSON для владельца — `ownerJson()`:**

```jsonc
{
  "id": 7,
  "code": "x9k2mp",
  "password": "",
  "uses_left": 1,          // -1 = без лимита
  "logins": ["ivan", "olga"],
  "enabled": true
}
```

Отдаётся владельцу списком в `/api/me/room/aliases`. `logins` в JSON —
массив строк (в базе тот же список лежит одной строкой, склейка/разбор — забота
SQLite-репозитория, не модели).

## Как этими моделями пользоваться

Модель — это «то, что ходит между слоями». Обычный поток данных на примере
входа пользователя:

```
network/HttpApi   → зовёт →   services/AuthService
                                     │  собирает объект User,
                                     │  проверяет пароль (сравнивает хеши)
                                     ▼
                              interface/IUsers::findByLogin()  → возвращает std::optional<User>
                                     │
                              interface/SqliteUsers            → SELECT … → строит User из строки БД
```

Ключевые точки:

- **Модель создаётся заполненной вручную** (в сервисе) и передаётся в
  репозиторий по ссылке: `void save(User &user)`. Если `user.id == -1` — это
  `INSERT`, и репозиторий проставит `user.id` после вставки; иначе — `UPDATE`.
- **Поиск возвращает `std::optional<Model>`.** Нет записи — `std::nullopt`,
  а не исключение и не «пустой объект». Всегда проверяй `has_value()` перед
  использованием.
- **Сервисы работают только с моделями и интерфейсами.** Они не знают, лежит
  User в SQLite или в памяти. Это и позволяет держать `InMemory`-реализацию
  для тестов рядом с боевой `sqlite`.

## Если ты пишешь новый интерфейс (репозиторий) под модель

Допустим, добавляешь хранилище для новой модели или новый бэкенд под
существующую. Мэппинг «модель ↔ таблица» делается в одном месте —
в SQLite-репозитории. Образец — `interface/sqlite/SqliteUsers.cpp`:

1. **Одна функция `rowToModel(stmt)`** собирает модель из строки результата
   (`colInt`, `colText`, `colBlob`, `colInt64` — по типу колонки). Она —
   единственное место, где порядок колонок в `SELECT` должен совпасть с
   порядком чтения.
2. **`save()` разветвляется по `id == -1`:** ветка `INSERT` (без `id`, потом
   `model.id = lastInsertId()`) и ветка `UPDATE … WHERE id = ?`.
3. **Соответствие полей и колонок** задаётся схемой в
   `interface/sqlite/SqliteDb.cpp` (`kSchema`). Поле `passSalt`/`passHash`
   (`QByteArray`) ↔ колонка `BLOB`; `QString` ↔ `TEXT`; `qint64` ↔ `INTEGER`;
   `bool` ↔ `INTEGER` (0/1). `QStringList` (`RoomAlias.logins`) в базе — одна
   `TEXT`-колонка: репозиторий склеивает список при записи и разбирает при
   чтении.

Правило, которое нельзя нарушать: **весь SQL живёт в `interface/sqlite/`**.
Модель не содержит ни строчки SQL и ничего не знает про базу — она и в
`InMemory`, и в `sqlite` абсолютно одинакова.

## Куда расти

Новые сущности (например, персистентная история чата) добавляются так:

1. Новый `struct` в `models/` (+ метод `*Json()`, если поедет к клиенту).
2. Колонки/таблица в `interface/sqlite/SqliteDb.cpp` (`kSchema`,
   `CREATE TABLE IF NOT EXISTS …` — переживает рестарт на готовой базе).
3. Интерфейс в `interface/Abstract/` + реализации `InMemory`/`sqlite`.
4. Бизнес-правила — в [`../services`](../services/README.md).

Смотри также: [`../interface`](../interface/README.md) (как хранится),
[`../services`](../services/README.md) (какие правила поверх),
[`../network`](../network/README.md) (как модели превращаются в HTTP/WS-ответы).
