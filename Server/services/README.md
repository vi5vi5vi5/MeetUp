# services/

**Бизнес-логика — «мозг» сервера.** Здесь живут все правила: как проверить
пароль, можно ли занять этот код комнаты, кому разрешён вход по ссылке.
Всё, что сложнее «положить и достать», происходит тут.

Читать после [`../models`](../models/README.md) (чем оперируем) и
[`../interface`](../interface/README.md) (откуда берём данные). Сервис —
это прослойка между сетью и хранилищем:

```
network/   ← разбирает HTTP/WS-запрос, вызывает метод сервиса, пакует ответ
   │
   ▼
services/  ← ПРАВИЛА: валидация, хеширование, права доступа, генерация кодов
   │
   ▼
interface/ ← кладёт/достаёт модели (SQLite или InMemory)
```

## Что делает сервис — и чего он НЕ делает

**Делает:** валидирует вход, применяет бизнес-правила, хеширует пароли,
генерирует коды, обновляет модели и сохраняет их через репозитории.

**Не делает трёх вещей — это важно:**

1. **Не знает про HTTP и WebSocket.** Принимает обычные значения (`QString`,
   `int`) или модели, возвращает результат или машинный код ошибки. Ни
   `HttpRequest`, ни сокетов, ни JSON здесь нет — это забота
   [`../network`](../network/README.md).
2. **Не пишет SQL.** Доступ к данным — только через интерфейсы
   `IUsers`/`ISessions`/… из [`../interface`](../interface/README.md).
3. **Не решает, кто вызывающий.** Сервис получает `ownerId`/`token` как
   аргумент и доверяет ему. Кто на самом деле сидит за запросом (по сессии из
   куки) — устанавливает сетевой слой до вызова.

Зависимости сервис получает через конструктор (`std::shared_ptr` на
репозитории), сам ничего глобального не создаёт. Точка сборки — `main.cpp`.

## Два сервиса

| Сервис | За что отвечает | Модели |
|---|---|---|
| `AuthService` | Регистрация, вход, сессии, профиль (имя, версия аватара), хеширование паролей | `User`, `Session` |
| `PersonalRoomService` | Личные комнаты и alias-ссылки: создание, настройка, удаление, правила входа | `PersonalRoom`, `RoomAlias` |

## Сквозной паттерн №1: результат-или-код-ошибки

Методы, которые могут «не получиться по бизнес-причине», возвращают не
`bool` и не бросают исключение, а **структуру-результат**: `AuthResult`,
`RoomResult`, `AliasResult`. Устроены одинаково:

```cpp
struct RoomResult {
    bool    ok = false;
    QString error;        // машинный код: "code_taken", "invalid_title", …
    PersonalRoom room;    // заполнено при ok == true

    static RoomResult fail(const char *code);   // краткий конструктор ошибки
};
```

**Код ошибки — машинный, не текст для человека.** `"login_taken"`, а не
«Логин занят». Дальше по цепочке:

```
services/  → "code_taken"
   │
network/HttpApi::statusForError()  → HTTP 409
   │
web/ (клиент)  → "Этот код уже занят"   ← человеческий текст ЖИВЁТ ТОЛЬКО ТУТ
```

Так текст ошибок не размазан по слоям, а лежит в одном месте — на странице,
рядом с остальным UI. Для того, кто пишет **клиент**, это значит: сервер
пришлёт `{"error": "code_taken"}`, и клиент сам маппит код → фразу. Полный
список кодов у каждого сервиса — в комментарии к его `*Result` в `.h`.

**Справочник кодов ошибок:**

| Сервис | Коды |
|---|---|
| `AuthService` → `AuthResult` | `invalid_login`, `invalid_display_name`, `weak_password`, `login_taken`, `wrong_credentials`, `no_session` |
| `PersonalRoomService` → `RoomResult` | `room_exists`, `invalid_code`, `code_taken`, `invalid_title`, `invalid_password`, `no_room` |
| `PersonalRoomService` → `AliasResult` | `no_room`, `no_alias`, `alias_limit`, `invalid_password`, `invalid_uses`, `invalid_logins` |

## Сквозной паттерн №2: нормализация → валидация → правило

Каждая операция создания/изменения проходит один и тот же конвейер, и порядок
важен:

1. **Нормализация** — привести вход к каноничному виду. `normalizeLogin` /
   `normalizeCode`: `trimmed().toLower()`. Логин `" Vi5 "` и `"vi5"` — один и
   тот же пользователь.
2. **Валидация** — формальная проверка формы (`validLogin`, `validCode`,
   `validTitle`, `validPassword`). Это `static`-методы: не зависят от
   состояния, их же можно переиспользовать где угодно. Правила зашиты в
   регулярку или простую проверку длины (напр. код: `^[a-z0-9_-]{3,32}$`).
3. **Бизнес-правило** — уникальность, лимиты, права. `code_taken` (код уже
   занят другой комнатой ИЛИ алиасом), `room_exists` (у пользователя уже есть
   комната), `alias_limit` (больше `kMaxAliases`).

Только пройдя все три, метод собирает модель, ставит `createdAtMs` и зовёт
`save()`. Именно инварианты, которые здесь гарантируются, [`../models`](../models/README.md)
описывает в комментариях к полям («нормализован: lowercase, 3–32») — а
обеспечиваются они **здесь**, не в модели.

## `AuthService` — аккаунты и сессии

Отвечает за регистрацию, вход, жизнь сессии и профиль. Пароли аккаунтов
хранятся только как `PBKDF2-HMAC-SHA512` (соль на пользователя); сам пароль
восстановить нельзя.

**Методы:**

| Метод | Что делает |
|---|---|
| `registerUserAsync(login, password, displayName, done)` | Создаёт аккаунт + сессию. **Асинхронный** (см. ниже). |
| `loginAsync(login, password, done)` | Проверяет пароль, выдаёт сессию. **Асинхронный.** |
| `logout(token)` | Удаляет сессию по токену. |
| `userByToken(token)` | Пользователь по токену куки. Побочно — **скользящее продление** сессии. |
| `changeDisplayName(token, name)` | Меняет отображаемое имя. |
| `setAvatarVer(token, ver)` | Обновляет счётчик версии аватара (сам файл кладёт HTTP-слой). |
| `purgeExpiredSessions()` | Чистка протухших сессий; зовётся по таймеру из `main`. |

### Почему register/login — асинхронные

Это самая нетривиальная часть слоя, и её стоит понять.

PBKDF2 **намеренно дорогой** (64000 итераций, ~0.1–0.3 с) — чтобы перебор
утёкших хешей был невыгоден. Но 0.3 с — это вечность для однопоточного event
loop: пока считается хеш, сервер **не мог бы релеить видео** другим участникам.

Решение — считать хеш в пуле потоков (`hashInPool`):

```
главный поток:  loginAsync() ─┐
                              │  отдаёт тяжёлый расчёт в пул и сразу выходит
пул потоков:        [ PBKDF2, 0.3 с ]   ← event loop в это время свободен
                              │
главный поток:  done(result) ◄┘  результат возвращается сюда (QueuedConnection)
```

Поэтому эти два метода принимают колбэк `done`, а не возвращают значение:
результат приходит **позже** и **обязательно в главном потоке** (репозитории
не потокобезопасны, трогать их из пула нельзя). Остальные методы синхронные —
они не считают хеши.

### Тонкости безопасности (заложены намеренно)

- **Вход неизвестного логина всё равно считает хеш** (по фиктивной соли) —
  иначе по мгновенному отказу можно было бы перечислять занятые логины
  (timing-оракул).
- **Сравнение хешей — за константное время** (`constantTimeEquals`), а не
  обычным `==`, который выдаёт длину совпавшего префикса по времени ответа.
- **Rehash при входе:** если глобальную стоимость `kPassIters` подняли, пароль
  тихо пере-хешируется с новой стоимостью, пока он в руках. Пользователя это не
  задерживает.
- **Скользящее продление:** сессия живёт 30 дней (`kSessionTtlMs`), но при
  каждом обращении, если осталось меньше половины, продлевается. Активный
  пользователь не разлогинится никогда, ушедший — через месяц.

## `PersonalRoomService` — комнаты и ссылки

Личные комнаты (одна на пользователя) и alias-ссылки на них. Все методы
синхронные. Про HTTP/WebSocket не знает; кто владелец — решает вызывающий,
передавая `ownerId`.

**Комнаты:**

| Метод | Что делает |
|---|---|
| `create(ownerId, code, title, password)` | Создать комнату (одна на пользователя). |
| `update(ownerId, code?, title?, password?)` | Частичное обновление (см. паттерн PATCH ниже). |
| `remove(ownerId)` | Удалить свою комнату. |
| `byOwner / byCode / byId` | Чтение (нормализуют код при поиске). |

**Alias-ссылки:**

| Метод | Что делает |
|---|---|
| `createAlias(ownerId, password, usesLeft, logins, enabled)` | Новая ссылка (до `kMaxAliases = 5` на комнату). Код генерирует сервер. |
| `updateAlias(ownerId, aliasId, password?, usesLeft?, logins?, enabled?)` | Частичное обновление. |
| `removeAlias(ownerId, aliasId)` | Удалить ссылку. |
| `aliasesByOwner / aliasByCode` | Чтение. |
| `consumeAlias(alias)` | Списать одно использование после успешного входа. |

### Сквозной паттерн №3: частичное обновление через `std::optional`

`update`/`updateAlias` реализуют **PATCH-семантику**: `std::nullopt` для
поля значит «не трогать», а не «очистить». Смотри сигнатуру:

```cpp
RoomResult update(int ownerId,
                  const std::optional<QString> &code,      // nullopt — код не меняем
                  const std::optional<QString> &title,     // nullopt — название не меняем
                  const std::optional<QString> &password); // nullopt — пароль не меняем;
                                                            // "" — УБРАТЬ пароль
```

Внутри каждое поле проверяется отдельно: `if (code.has_value()) { … }`.
Так один метод покрывает и «сменить только название», и «сменить всё сразу».
Пустая строка ≠ nullopt: `password = ""` означает осознанное «убрать пароль».

### Правило безопасности: всё owner-scoped

Любая операция с алиасом проверяет, что он принадлежит комнате **именно этого**
владельца:

```cpp
if (!room.has_value() || existing->roomId != room->id)
    return AliasResult::fail("no_alias");   // чужой id неотличим от несуществующего
```

Клиент присылает `aliasId`, но не может тронуть чужую ссылку — сервис сверяет
владельца по `ownerId`. Чужой/несуществующий id дают один и тот же ответ
(`no_alias`), чтобы нельзя было по ответу узнать, существует ли такой алиас.

### Тонкости

- **Генерация кода алиаса** (`generateAliasCode`): 10 случайных символов из 36
  от криптостойкого источника, коллизии с комнатами и алиасами
  перегенерируются. Подобрать перебором нельзя.
- **`consumeAlias` перечитывает счётчик** из базы перед списанием: параллельные
  входы могли его уже подвинуть. Дойдя до 0, алиас удаляется — так работает
  «одноразовая ссылка». Безлимитные (`usesLeft == -1`) не трогаются.
- **`normalizeLogins`** для белого списка: trim + lowercase + выброс пустых и
  дублей; вернёт `nullopt` (→ `invalid_logins`), если логинов >50 или логин
  длиннее 64/содержит запятую (запятая — разделитель в хранилище, см.
  [`../interface`](../interface/README.md)).

## Как отсюда растить новые фичи

Новая бизнес-возможность (поиск сообщений, реакции, персистентный чат) —
это новый сервис или новый метод существующего, по тем же паттернам:

1. Метод принимает простые значения/модели, возвращает `*Result` с машинным
   кодом ошибки (не текст, не исключение).
2. Проходит конвейер нормализация → валидация (`static`-методы) → бизнес-правило.
3. Данные — только через интерфейс из [`../interface`](../interface/README.md).
4. Зависимости — через конструктор, объект создаётся в `main.cpp`.
5. Долгий расчёт (крипта и т.п.) — асинхронно через пул, колбэк в главном потоке.

Как эти методы вызываются из сети и превращаются в HTTP/WS — в
[`../network`](../network/README.md).
