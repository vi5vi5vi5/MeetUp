# interface/

**Слой доступа к данным.** Здесь живёт ответ на вопрос «где и как хранятся
модели». Приём взят из [MedFlow](https://github.com/vi5vi5vi5/medflow):
контракт хранилища отделён от его реализации, поэтому бизнес-логика из
[`../services`](../services/README.md) не знает, лежат ли данные в SQLite или
в оперативной памяти.

Читать эту папку стоит **после** [`../models`](../models/README.md): интерфейсы
только принимают и возвращают модели, ничего к ним не добавляя.

## Идея слоя: контракт отдельно, реализация отдельно

```
services/            ← работают ТОЛЬКО с интерфейсами (IUsers, ISessions, …)
   │  зависят от абстракции, не от реализации
   ▼
interface/Abstract/  ← чистые интерфейсы: «что храним и как ищем»
   ▲            ▲
   │            │  один и тот же контракт — две реализации
InMemory/     sqlite/
(в памяти)    (файл на диске)  ← «где именно храним»
```

Сервис держит `std::shared_ptr<IUsers>` и вызывает `findByLogin(...)`. Какой
объект стоит за указателем — `InMemoryUsers` или `SqliteUsers` — решается
**один раз в `main.cpp`** при сборке зависимостей. Сервис этого не видит. Это
классический **Dependency Inversion**: и высокий слой (сервис), и низкий
(хранилище) зависят от абстракции посередине.

Зачем так:
- **Реализацию можно подменить** без единой правки в сервисах. Сейчас боевая —
  `sqlite`; `InMemory` осталась как эталон и для быстрых тестов без файла на
  диске.
- **Сервисы тестируются в изоляции**: подсунул `InMemory`-репозиторий — и
  проверяешь бизнес-правила без базы.

## Три подпапки

| Папка | Что внутри | Роль |
|---|---|---|
| `Abstract/` | Чистые интерфейсы `IUsers`, `ISessions`, `IPersonalRooms`, `IRoomAliases` | **Контракт.** Только `= 0`-методы, никакой реализации. |
| `sqlite/` | `SqliteDb` + `SqliteStmt` (обёртки над C-API) и по репозиторию на сущность | **Боевое хранение.** Весь SQL проекта — здесь и больше нигде. |
| `InMemory/` | По классу на сущность, данные в `QHash` | **Эталон/тесты.** Живёт до перезапуска сервера. |

> SQLite подключён как **амальгамация** (`third_party/sqlite/sqlite3.c`) — один
> `.c`-файл прямо в проекте, а не через Qt `QtSql`. Так у сервера нет внешней
> зависимости от драйвера БД. Собран с `SQLITE_THREADSAFE=0`: все обращения
> идут только из главного потока (как и к `InMemory`), блокировки не нужны.

## `Abstract/` — контракты

Каждый интерфейс — это чистый абстрактный класс: виртуальный деструктор +
набор `= 0`-методов. Реализации наследуют и переопределяют (`override`).

Общая форма у всех четырёх репозиториев одинаковая:

| Метод | Смысл | Возврат |
|---|---|---|
| `save(Model &)` | `id == -1` → вставить (проставит `id`); иначе обновить | `void` |
| `findById(int)` | Достать по первичному ключу | `std::optional<Model>` |
| `findBy…(…)` | Достать по другому полю (см. ниже) | `std::optional<Model>` |
| `list…(…)` | Достать несколько | `QList<Model>` |
| `removeBy…(…)` | Удалить | `bool` (было ли что удалять) |

Два железных правила:

1. **Поиск возвращает `std::optional`, а не бросает исключение.** Нет записи —
   `std::nullopt`. Вызывающий обязан проверить `has_value()`.
2. **Каждый `findBy…`, кроме `findById`, — это будущий индекс в SQLite.**
   Поэтому поиск по логину — отдельный метод `findByLogin`, а не абстрактный
   «предикат»: под него в схеме стоит `UNIQUE`-индекс. Интерфейс описывает
   *как мы ищем*, потому что от этого зависит структура таблицы.

Конкретные методы по сущностям:

| Интерфейс | Особые методы поиска | Комментарий |
|---|---|---|
| `IUsers` | `findByLogin` | Логин уже нормализован (`AuthService::normalizeLogin`) |
| `ISessions` | `findByToken`, `removeByUser`, `removeExpired` | Ключ — токен, не `id`. `removeByUser` — разлогин со всех устройств; `removeExpired` — чистка по расписанию |
| `IPersonalRooms` | `findByCode`, `findByOwner` | Комната одна на пользователя, поэтому поиск по владельцу однозначен |
| `IRoomAliases` | `findByCode`, `listByRoom` | `listByRoom` возвращает `QList` — алиасов у комнаты несколько |

## `sqlite/` — боевое хранение

Две служебные обёртки + репозитории:

**`SqliteDb`** — одно соединение с файлом базы. При создании открывает файл и
накатывает схему (`CREATE TABLE IF NOT EXISTS …` — переживает рестарт на уже
готовой базе). Раздаёт `handle()` репозиториям, умеет `lastInsertId()` и
`changes()`. Схема всех таблиц — в `SqliteDb.cpp` (`kSchema`), это единственное
место, где описана структура БД.

**`SqliteStmt`** — один подготовленный запрос на время одного вызова
репозитория. RAII: конструктор готовит SQL, деструктор освобождает. Интерфейс
маленький и его стоит запомнить, потому что все репозитории написаны через него:

```cpp
SqliteStmt q(*m_db, "SELECT id, login FROM users WHERE id = ?");
q.bind(1, id);          // параметры нумеруются с 1 (как в SQLite)
if (q.step()) {         // true — есть строка результата
    int uid = q.colInt(0);      // колонки нумеруются с 0
    QString login = q.colText(1);
}
```

- `bind(i, …)` — перегружен под `int`, `qint64`, `QString`, `QByteArray`.
- `step()` — `true`, пока есть строки; для `INSERT`/`UPDATE`/`DELETE` зовётся
  один раз, чтобы выполнить.
- `col…(i)` — читает колонку: `colInt`, `colInt64`, `colText`, `colBlob`.

### Анатомия репозитория (образец — `SqliteUsers.cpp`)

Каждый SQLite-репозиторий устроен по одному шаблону из трёх частей:

**1. `rowToModel(stmt)`** — единственное место, где строка результата
превращается в модель. Порядок чтения колонок обязан совпадать с порядком в
`SELECT`:

```cpp
User rowToUser(const SqliteStmt &q) {
    User u;
    u.id    = q.colInt(0);     // порядок ↔ порядок колонок в SELECT
    u.login = q.colText(1);
    // …
    return u;
}
```

**2. `save()` разветвляется по `id == -1`** — `INSERT` (без `id`, потом
`model.id = m_db->lastInsertId()`) либо `UPDATE … WHERE id = ?`.

**3. `findBy…()`** — `SELECT` с одним и тем же списком колонок, `bind`
параметра, `step()`, и либо `rowToModel`, либо `std::nullopt`.

### Когда типы модели и БД не совпадают напрямую

Иногда поле модели не ложится один-в-один на колонку. Тогда преобразование
делается **внутри репозитория** — модель об этом не знает. Каноничный пример —
`RoomAlias.logins` (`QStringList`) хранится одной `TEXT`-колонкой через запятую
(`SqliteRoomAliases.cpp`):

```cpp
// запись:  список → строка
const QString logins = alias.logins.join(QLatin1Char(','));
q.bind(5, logins);

// чтение:  строка → список
const QString logins = q.colText(5);
a.logins = logins.isEmpty() ? QStringList{} : logins.split(QLatin1Char(','));
```

Отдельная таблица под ≤50 логинов не окупается — поэтому склейка. Так же
`bool enabled` ↔ `INTEGER` (0/1). Это и есть граница «модель ↔ схема»: вся
такая мелочь живёт здесь, а не протекает в сервисы или модели.

## `InMemory/` — эталон и тесты

Те же интерфейсы, но данные — в `QHash` (образец — `InMemoryUsers.h`).
Реализация целиком в заголовке, потому что короткая. Полезные детали:

- **`id` раздаётся вручную** счётчиком `m_nextId` (в SQLite это делает
  `AUTOINCREMENT`).
- **Индексы делаются вторым `QHash`.** У `InMemoryUsers` есть `m_byLogin`
  (`login → id`), чтобы `findByLogin` не сканировал всех — ровно то, что в
  SQLite даёт `UNIQUE`-индекс. При переименовании/удалении оба хеша
  синхронизируются вручную.

Держать эту реализацию рядом с боевой полезно: она — живая проверка, что
интерфейс не протёк деталями SQLite (если метод невозможно реализовать в
памяти без SQL — значит, контракт спроектирован плохо).

## Как здесь всё собирается вместе (`main.cpp`)

Реализации выбираются в одном месте — точке сборки зависимостей. Чтобы
переехать с боевого SQLite на InMemory (например, для теста), меняются
только эти строки, всё остальное — нет:

```cpp
auto db    = std::make_shared<SqliteDb>(dbPath);
auto users = std::make_shared<SqliteUsers>(db);   // ← тут выбор реализации
auto auth  = std::make_shared<AuthService>(users, sessions);
//                                          └ сервис принимает shared_ptr<IUsers>,
//                                            конкретный тип ему безразличен
```

## Правила слоя

- Интерфейс описывает **что** храним и **как ищем**, но не **где**: ни строчки
  SQL, ни путей к файлам, ни глобальных переменных в `Abstract/`.
- Реализации **не содержат бизнес-логики**: валидация, хеширование паролей,
  правила доступа — это [`../services`](../services/README.md). Репозиторий
  только кладёт и достаёт.
- Экземпляры создаются в `main.cpp` и передаются сервисам через
  `std::shared_ptr`. Сам репозиторий ничего глобального не создаёт.

## Если ты добавляешь новую сущность (полный чек-лист)

1. Модель — в [`../models`](../models/README.md) (+ `*Json()`, если едет к
   клиенту).
2. Таблица — в `sqlite/SqliteDb.cpp`, блок `kSchema`
   (`CREATE TABLE IF NOT EXISTS …`).
3. Интерфейс — `Abstract/IThing.h`: `save` + нужные `findBy…` + `removeBy…`.
4. Реализация SQLite — `sqlite/SqliteThing.{h,cpp}` по шаблону из
   `SqliteUsers.cpp` (`rowToModel` + `save` с ветвлением + `findBy…`).
5. Реализация InMemory — `InMemory/InMemoryThing.h` (по желанию, но помогает
   держать контракт честным).
6. Сборка — добавить `.cpp` в `CMakeLists.txt` и создать объект в `main.cpp`.

Дальше — правила поверх данных: [`../services`](../services/README.md).
