# network/

Входные точки сервера — всё, что слушает порты, живёт здесь.

| Файл | За что отвечает |
|---|---|
| `ConferenceServer.*` | Обёртка над `QWebSocketServer`: принимает соединения, заворачивает каждое в `ClientSession`, разбирает JSON-сообщения (`join` / `chat`) и релеит бинарные медиа-кадры участникам комнаты. |
| `HttpFileServer.*` | HTTP-транспорт: принимает соединения, собирает запросы парсером, отдаёт статику веб-клиента из [`../web`](../web/README.md) (только GET, наружу из корневой папки не выпускает), а пути `/api/*` передаёт в `HttpApi`. |
| `HttpRequest.*` | `HttpRequest` — разобранный запрос (метод, путь, заголовки, куки, тело) и `HttpRequestParser` — инкрементальный сборщик: копит куски из сокета до полного запроса, лимиты: заголовки 16 КиБ, тело 1 МиБ. |
| `HttpApi.*` | Маршрутизация JSON API: `/api/auth/*` (регистрация, вход, выход), `/api/me` (профиль), `/api/rooms` (комнаты). Про сокеты не знает: получает `HttpRequest`, зовёт сервисы, возвращает `ApiResponse` (статус + JSON + Set-Cookie). |

Сессия входа передаётся HttpOnly-кукой `meetup_session` — токен недоступен
из JS, браузер шлёт его сам.

TLS этот слой не делает: в проде https/wss терминирует nginx из
[`../proxy`](../proxy/README.md), сюда трафик приходит уже расшифрованным.

## Куда расти

- Новые типы WebSocket-сообщений добавляются в `ConferenceServer::onText`:
  ветка `type == "..."` → свой `handleX()`, который разбирает JSON и зовёт
  сервис из [`../services`](../services/README.md).
- Новые HTTP-маршруты — ветка в `HttpApi::route` → свой `handleX()`. Правило
  то же: сетевой слой разбирает запрос и зовёт сервис, бизнес-логики здесь нет.
