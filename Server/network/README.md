# network/

**Входные точки сервера и его протокол.** Всё, что слушает порты, живёт здесь.
Для того, кто пишет клиент, это **главная опорная папка**: тут точно описано,
на какие адреса стучаться, какие запросы слать, что придёт в ответ и в каком
формате. Читать после [`../models`](../models/README.md) (формат данных) и
[`../services`](../services/README.md) (коды ошибок).

> Эта методичка описывает **сам слой и протокол**: эндпоинты, сообщения,
> форматы, статусы. Как из этого собрать полноценный клиент (выбор кодеков,
> E2E-шифрование, джиттер-буфер, скелет на Qt6) — в отдельном большом гайде
> [`docs/desktop-client-guide.md`](../docs/desktop-client-guide.md). Здесь на
> него ссылки там, где тема выходит за рамки слоя.

## Два независимых сервера, два порта

`network` поднимает **две** несвязанные точки входа. Клиент работает с обеими:

| Сервер | Порт (лок.) | Протокол | Для чего |
|---|---|---|---|
| `HttpFileServer` | **80** (`--http-port`) | HTTP/1.1 | Статика веб-клиента + JSON API (`/api/*`) |
| `ConferenceServer` | **9000** (`--ws-port`) | WebSocket | Живая конференция: вход в комнату, чат, медиа |

Оба реестр комнат (`RoomRegistry`) держат общий — HTTP создаёт комнаты, WS в
них подключает. В проде оба порта прячутся за nginx на 443 (см. про TLS ниже).

**Про TLS:** этот слой работает по **чистому http/ws**, без шифрования канала.
В проде https/wss терминирует nginx из [`../proxy`](../proxy/README.md), сюда
трафик приходит уже расшифрованным. Значит, клиент в проде ходит на
`https://<домен>/` и `wss://<домен>/ws`, локально — `http://localhost` и
`ws://localhost:9000`.

## Четыре файла слоя

| Файл | Роль |
|---|---|
| `HttpFileServer.*` | HTTP-транспорт: TCP-соединения, отдача статики, делегирование `/api/*` в `HttpApi` |
| `HttpRequest.*` | `HttpRequest` (разобранный запрос) + `HttpRequestParser` (инкрементальный сборщик из TCP-кусков) |
| `HttpApi.*` | Маршрутизация `/api/*`: разбирает запрос, зовёт сервисы, собирает `ApiResponse`. Бизнес-логики нет — только перевод HTTP ↔ вызовы сервисов |
| `ConferenceServer.*` | WebSocket-ядро: вход в комнаты, чат, релей медиа |

Принцип, общий для всего слоя: **`network` разбирает/собирает протокол, но не
принимает бизнес-решений.** Проверить пароль, занять код, списать использование
ссылки — это [`../services`](../services/README.md). Слой лишь переводит
«байты из сокета» ↔ «вызов сервиса».

# Часть 1. HTTP — как устроен транспорт

Прежде чем перечислять эндпоинты — как запрос вообще обрабатывается, потому что
от этого зависят ограничения клиента.

### Путь запроса

```
TCP-соединение → onReadyRead → HttpRequestParser.feed(chunk)
   │  копит байты, пока не соберёт весь запрос (заголовки + тело)
   ▼
dispatch:
   ├─ путь в /api/*  → HttpApi.route(...) → сервис → ApiResponse
   └─ иначе (GET)    → serveFile(...) отдаёт файл из web/
   ▼
sendResponse: HTTP/1.1 <code> … + Connection: close  (соединение закрывается)
```

### Что клиент обязан учитывать

- **Одно соединение = один запрос.** Каждый ответ идёт с `Connection: close`,
  keep-alive и pipelining не поддерживаются. Хвост после тела игнорируется.
- **Тело — только по `Content-Length`.** `Transfer-Encoding: chunked` **не
  поддерживается** (ответ `501`). `fetch` с готовым телом и `curl` шлют
  `Content-Length` сами — проблем нет; в нативном клиенте выставляй его явно.
- **Лимиты парсера** (`HttpRequestParser`):
  - заголовки > **16 КиБ** → `431 Request Header Fields Too Large`;
  - тело > **1 МиБ** → `413 Content Too Large`.
- **Путь декодируется** из percent-encoding, query-строка (`?...`) отбрасывается
  до маршрутизации (но читается как есть в URL, напр. `?v=` для аватарки — это
  для браузерного кэша, сервер её не парсит).

### Формат ответа

Всякий ответ — `HTTP/1.1`, с заголовками `Content-Type`, `Content-Length`,
`Cache-Control`, `Connection: close`. JSON-ответы API — всегда
`application/json; charset=utf-8` и `Cache-Control: no-store`.

### Отдача статики (`serveFile`)

- `/` → отдаёт `login.html` (точка входа продукта).
- Путь нормализуется и **не выпускается за пределы `web/`** (защита от
  `../` — иначе `403`).
- `/assets/*` (библиотеки, шрифты) кэшируются (`max-age=86400`), страницы —
  `no-store` (всегда свежие).
- MIME определяется по расширению; неизвестное — `application/octet-stream`.

# Часть 2. HTTP API — полный справочник эндпоинтов

Все пути начинаются с `/api`. Общие правила:

- **Тело запроса и ответа — JSON.** Невалидный JSON в теле → `400 invalid_json`.
- **Ошибки** — всегда `{"error": "<машинный_код>"}` + соответствующий HTTP-статус.
  Расшифровка кодов — в [`../services`](../services/README.md); маппинг код →
  статус делает `HttpApi::statusForError` (`*_taken`/`*_exists`/`alias_limit` →
  409; `wrong_credentials`/`no_session` → 401; `no_room`/`no_alias` → 404;
  прочее → 400). Неизвестный метод на существующем пути → `405`.
- **Аутентификация — кукой** `meetup_session` (см. раздел про сессии). Эндпоинты,
  где нужен вход, при её отсутствии/протухании возвращают `401 no_session`.

Легенда: 🔓 — без входа, 🔒 — нужна кука сессии.

## 2.1. Аккаунты и сессия

### 🔓 `POST /api/auth/register` — регистрация

Запрос:
```json
{ "login": "vi5", "password": "hunter2!!", "display_name": "Пётр" }
```
Успех `200` — заводит аккаунт, сразу логинит (ставит куку):
```json
{ "user": { "id": 42, "login": "vi5", "display_name": "Пётр", "avatar_ver": 0 } }
```
+ заголовок `Set-Cookie: meetup_session=<токен>; Path=/; HttpOnly; SameSite=Lax; Max-Age=…`

Ошибки: `invalid_login`, `invalid_display_name`, `weak_password` (400),
`login_taken` (409). **Асинхронный**: пароль хешируется в пуле потоков, ответ
приходит с задержкой ~0.1–0.3 с — это норма (см. `AuthService` в
[`../services`](../services/README.md)).

### 🔓 `POST /api/auth/login` — вход

Запрос: `{ "login": "vi5", "password": "hunter2!!" }`
Успех `200` — та же форма, что у register (`{"user": …}` + `Set-Cookie`).
Ошибка: `wrong_credentials` (401). Тоже асинхронный.

### 🔒 `POST /api/auth/logout` — выход

Тело не нужно. Всегда `200`, гасит куку (`Max-Age=0`), даже если сессии не было.

### 🔒 `GET /api/me` — кто я

Успех `200`: `{ "user": { … } }`. Нет валидной сессии → `401 no_session`.
Побочно продлевает сессию (скользящее окно).

### 🔒 `PATCH /api/me` — сменить имя

Запрос: `{ "display_name": "Новое имя" }`
Успех `200`: `{ "user": { … } }`. Ошибки: `nothing_to_update` (нет поля),
`invalid_display_name`, `no_session`.

## 2.2. Аватарки

Аватар — один JPEG-файл на пользователя (`<data>/avatars/<id>.jpg`), новая
загрузка перезаписывает старую (истории нет).

### 🔒 `POST /api/me/avatar` — загрузить/заменить

Запрос: `{ "image": "<base64 JPEG>" }`. Требования: валидный JPEG (проверяется
сигнатура `FF D8 FF`), декодированный размер ≤ **256 КиБ**. Успех `200`:
`{ "user": { … } }` с увеличенным `avatar_ver`. Ошибка: `bad_image` (400).

### 🔒 `DELETE /api/me/avatar` — удалить

Удаляет файл, сбрасывает `avatar_ver` в 0. Успех `200`: `{ "user": { … } }`.

### 🔓 `GET /api/users/<id>/avatar` — публичная раздача

Отдаёт **сам JPEG** (`Content-Type: image/jpeg`), не JSON. Нет аватарки →
`404 no_avatar`. Кэш вечный (`immutable`), поэтому клиент запрашивает с
версией: **`/api/users/42/avatar?v=3`**. Меняется аватар → растёт `avatar_ver`
→ новый URL → браузер берёт свежую картинку. `?v=` сервер не читает — это только
для кэша.

## 2.3. Разовые комнаты

### 🔓 `POST /api/rooms` — создать разовую конференцию

Тело не нужно. Успех `200`: `{ "room": "<КОД24>" }` — случайный код из 24
символов. Именно так создаётся обычная конференция: код-ссылка и есть её секрет.

### 🔓 `GET /api/rooms/<code>` — проверить комнату перед входом

Клиент зовёт до WS-join, чтобы узнать тип комнаты и нужен ли пароль. Три случая
ответа:

**Разовая комната в эфире:**
```json
{ "room": "<code>", "participants": 3 }
```
**Личная комната** (существует в базе, может быть офлайн):
```json
{ "room": "<code>", "personal": true, "title": "Переговорка",
  "has_password": true, "online": true, "participants": 2 }
```
**Не найдена** (ни живой разовой, ни личной, ни активного алиаса) → `404 room_not_found`.

> Если `<code>` — это alias-ссылка, ответ ведёт в её личную комнату, но
> `"room"` содержит **код алиаса** (join пойдёт по нему, чтобы применились
> правила ссылки), а `has_password` — про пароль **алиаса**, не комнаты.

## 2.4. Личная комната владельца 🔒

Всё под `/api/me/room` — операции владельца над своей единственной комнатой.

### `GET /api/me/room` — моя комната

`200`: `{ "room": { "code", "title", "password", …live-инфо } }` — владельцу
отдаётся **всё, включая пароль** (`ownerJson`), плюс живое состояние:
`online`, `participants`, `participant_names` (массив имён), `live_since_ms`
(старт эфира). Нет комнаты → `404 no_room`.

### `POST /api/me/room` — создать

Запрос: `{ "code": "vi5", "title": "Переговорка", "password": "" }`
`200`: та же форма, что GET. Ошибки: `room_exists` (уже есть комната),
`invalid_code`, `code_taken`, `invalid_title`, `invalid_password`.

### `PATCH /api/me/room` — изменить (частично)

**PATCH-семантика**: присылаешь только меняемые поля; отсутствие поля = «не
трогать»; `"password": ""` = убрать пароль.
```json
{ "title": "Новое название" }
```
`200`: обновлённая комната. Ошибки те же, что у POST.

### `DELETE /api/me/room` — удалить

`200` (пустой ответ). Нет комнаты → `404 no_room`.

### `POST /api/me/room/close` — завершить эфир

Выгоняет **всех** участников (каждому шлётся WS `{"type":"error","reason":"room_closed"}`,
сокет закрывается). `200`: `{ "kicked": <число> }`. Сама комната-настройка
остаётся, удаляется только живой эфир.

## 2.5. Alias-ссылки комнаты 🔒

Дополнительные ссылки со своими правилами (до 5 на комнату). Подробно про поля —
[`RoomAlias` в models](../models/README.md).

### `GET /api/me/room/aliases` — список
`200`: `{ "aliases": [ { "id", "code", "password", "uses_left", "logins", "enabled" }, … ] }`

### `POST /api/me/room/aliases` — создать
Запрос (все поля необязательны):
```json
{ "password": "", "uses": 1, "logins": ["ivan"], "enabled": true }
```
`uses`: число (лимит входов), `null` (без лимита), отсутствует (без лимита).
`200`: `{ "alias": { … } }` со сгенерированным сервером `code`. Ошибки:
`no_room`, `alias_limit` (409, уже 5), `invalid_password`, `invalid_uses`,
`invalid_logins`.

### `PATCH /api/me/room/aliases/<id>` — изменить (частично)
Те же поля, PATCH-семантика. `200`: `{ "alias": { … } }`. Чужой/несуществующий
`id` → `404 no_alias`.

### `DELETE /api/me/room/aliases/<id>` — удалить
`200` (пусто) либо `404 no_alias`.

# Часть 3. Сессия и кука — как держать вход

Единственный способ аутентификации во всём API и в WS — кука `meetup_session`.

- **Выдаётся** в `Set-Cookie` при register/login. Атрибуты:
  `Path=/; HttpOnly; SameSite=Lax; Max-Age=<30 дней>`. `Secure` не ставится
  (локально http; в проде куку доносит https-прокси).
- **`HttpOnly`** — из JavaScript токен недоступен; браузер шлёт куку сам.
  **Для нативного клиента (Qt/Android):** после login сохрани значение
  `Set-Cookie` и прикладывай заголовок `Cookie: meetup_session=<токен>` к
  каждому последующему запросу — и к HTTP, и к WS-хендшейку. Парсить токен не
  надо, он непрозрачный.
- **Продление скользящее:** активный клиент не разлогинится; сессия живёт 30
  дней от последнего обращения.

# Часть 4. WebSocket — вход в конференцию и обмен

Второй сервер (`ConferenceServer`, порт 9000 / `wss://…/ws` в проде). После
установки WS-соединения клиент и сервер обмениваются двумя видами сообщений:

- **Текстовые (JSON)** — управление: вход, чат, состояние, демонстрация.
- **Бинарные** — медиа-кадры (сервер их не декодирует, только релеит).

**Привязка к аккаунту при хендшейке:** если в WS-запросе есть кука
`meetup_session`, сервер узнаёт пользователя. Авторизованный получает
**постоянный id** (`0x80000000 + userId`) и имя из профиля; аноним — очередной
id со счётчика и имя из своего `join`. Один и тот же аккаунт входит в любую
конференцию под одним id.

## 4.1. Клиент → сервер (JSON)

Каждое сообщение — объект с полем `type`.

### `join` — войти в комнату
```json
{ "type": "join", "room": "<code>", "name": "Гость", "password": "..." }
```
- `name` — для анонима (у авторизованного берётся из профиля, поле игнорируется).
- `password` — если требуется (личная комната/алиас с паролем).
- В ответ — `join_ok` (успех) либо `error` (см. коды ниже). Повторный `join`
  на том же соединении → `error already_joined`.

### `chat` — сообщение в чат
```json
{ "type": "chat", "text": "привет", "image": "<base64 JPEG>" }
```
- Любое из полей необязательно, но не оба пустые. `text` обрезается до 2000
  символов; `image` (base64) ≤ **600000** символов, иначе `error image_too_large`.
- Рассылается **всем, включая отправителя** (единая лента), с временной меткой
  сервера. Не в комнате → `error not_in_room`.

### `state` — моё состояние медиа
```json
{ "type": "state", "mic": true, "cam": false }
```
Поля необязательны (меняется только присланное). Рассылается остальным как
`participant_state` — чтобы рисовать заглушку/значок mic-off.

### `screen` — демонстрация экрана
```json
{ "type": "screen", "on": true }
```
`on:true` — застолбить демонстрацию за собой (одна на комнату; занято →
`error screen_busy`), `on:false` — отпустить. Подтверждение — сообщение
`screen` всем.

### `ping` — замер задержки
```json
{ "type": "ping", "t": 1720000000000 }
```
Сервер возвращает `{"type":"pong","t":<та же метка>}`. RTT считает клиент.

## 4.2. Сервер → клиент (JSON)

### `join_ok` — вход состоялся
```json
{ "type": "join_ok", "sender_id": 42,
  "participants": [ { "id", "name", "mic", "cam", "user_id"?, "avatar"? }, … ],
  "history": [ { "sender_id", "sender_name", "text", "timestamp_ms", "image"?, "image_dropped"? }, … ],
  "room_title": "Переговорка",   // только для личной комнаты
  "screen_id": 7 }               // если кто-то уже демонстрирует экран
```
`sender_id` — твой id в этой комнате. `participants` — кто уже внутри,
`history` — чат до твоего входа (последние ≤500 сообщений; у картинок старше
24-й приходит `image_dropped:true` вместо данных). `user_id`+`avatar` есть у
авторизованных — по ним клиент строит URL аватарки.

### `participant_joined` / `participant_left`
```json
{ "type": "participant_joined", "id": 7, "name": "Гость",
  "mic": true, "cam": true, "user_id"?: 15, "avatar"?: 2 }
{ "type": "participant_left", "id": 7 }
```

### `participant_state` — сосед сменил mic/cam
```json
{ "type": "participant_state", "id": 7, "mic": false, "cam": true }
```

### `chat` — новое сообщение (в т.ч. своё)
```json
{ "type": "chat", "sender_id": 7, "sender_name": "Гость",
  "text": "привет", "timestamp_ms": 1720000000000, "image"?: "<base64>" }
```

### `screen` — сменился статус демонстрации
```json
{ "type": "screen", "id": 7, "on": true }
```

### `pong` — ответ на ping
```json
{ "type": "pong", "t": 1720000000000 }
```

### `error` — отказ/событие
```json
{ "type": "error", "reason": "wrong_password" }
```
Коды `reason`: `invalid_json`, `unknown_type`, `invalid_join`, `already_joined`,
`room_not_found`, `room_offline` (личная комната ещё не открыта владельцем),
`wrong_password`, `alias_forbidden` (логин не в списке допущенных),
`not_in_room`, `image_too_large`, `screen_busy`, `session_replaced` (тебя
вытеснил новый вход с того же аккаунта — **не переподключайся**), `room_closed`
(владелец завершил эфир).

## 4.3. Бинарный протокол (медиа) — кратко

Медиа-кадры (видео/аудио) ходят **бинарными** WS-сообщениями. Сервер их **не
декодирует и не понимает формат** — это SFU-релей: принял кадр от одного,
разослал остальным в комнате.

Единственное, что сервер делает с байтами: **вставляет `sender_id`** после
первого байта. Клиент шлёт заголовок `[type:1][timestamp_ms:8][payload]`,
получатели видят `[type:1][sender_id:4 LE][timestamp_ms:8][payload]`. Так все
знают, чей это кадр, а отправителю его не возвращают.

Ограничения на сервере: сообщение короче 9 байт отбрасывается; кадры
демонстрации экрана (`type` 5/7) ретранслируются **только** от того, за кем
закреплён `screen`. Лимит входящего WS-сообщения — 2 МиБ (см.
[`../core`](../core/README.md), `ClientSession`).

> Полная спецификация бинарного формата, номера типов кадров, кодеки, E2E,
> джиттер-буфер и готовый код упаковки/распаковки на Qt — в
> [`docs/desktop-client-guide.md`](../docs/desktop-client-guide.md), разделы 5–7.
> Здесь описано только то, что делает с кадром сам сервер.

# Часть 5. Типичные сценарии (склейка HTTP + WS)

**Анонимный вход в конференцию по коду:**
```
GET /api/rooms/<code>        → узнать тип и нужен ли пароль
открыть WS ws://…:9000
→ {"type":"join","room":"<code>","name":"Гость","password":"…"}
← join_ok (участники + история)
← participant_joined (когда входят другие)
```

**Регистрация и вход авторизованного:**
```
POST /api/auth/register      → Set-Cookie meetup_session
(кука прикладывается ко всем следующим запросам и к WS-хендшейку)
открыть WS с кукой → join без name/password → id и имя берутся из профиля
```

**Владелец открывает личную комнату:**
```
POST /api/me/room {code,title,password}   → создать (однажды)
WS join room=<свой code> (с кукой)        → сервер поднимает эфир (createPersonal)
… гости теперь могут войти (room_offline пропадает)
POST /api/me/room/close                   → завершить, выгнать всех
```

## Куда расти

- **Новый HTTP-маршрут:** ветка в `HttpApi::route` → свой `handleX()`. Правило —
  разобрать запрос, позвать сервис из [`../services`](../services/README.md),
  вернуть `ApiResponse`. Бизнес-логики в хендлере нет.
- **Новый тип WS-сообщения:** ветка `type == "..."` в `ConferenceServer::onText`
  → свой `handleX(session, msg)`. Так же: разобрать JSON, позвать сервис/комнату.
- **Новый вид ответа:** `ApiResponse` умеет и JSON (`body`), и бинарь
  (`rawBody`+`contentType`, как аватарки) — новый бинарный ответ не требует
  трогать транспорт.

управляет: сессии, комнаты), [`../services`](../services/README.md) (куда слой
делегирует решения), [`docs/desktop-client-guide.md`](../docs/desktop-client-guide.md)
(как из этого протокола собрать клиент целиком).
