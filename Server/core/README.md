# core/

**Модель времени выполнения сервера — «живое» состояние.** Здесь — то, чего
нет в базе: кто прямо сейчас подключён, в какой он комнате, кто с кем говорит.
Всё это существует только в оперативной памяти и исчезает при перезапуске
сервера.

Разница с нижними слоями принципиальная:

| Слой | Что хранит | Живёт |
|---|---|---|
| `models`/`interface` | Аккаунты, комнаты-настройки | в базе, между перезапусками |
| **`core`** | **Соединения, кто в эфире, история чата эфира** | **в памяти, до перезапуска** |

`PersonalRoom` из [`../models`](../models/README.md) — это *настройка* комнаты
(код, пароль, название) в базе. `ConferenceRoom` из `core` — это *живой эфир*
той же комнаты в памяти: список подключённых сокетов. Не путать: одна и та же
комната имеет обе стороны.

Этот слой **не слушает порты и не разбирает протокол** — этим занимается
[`../network`](../network/README.md). `core` — это объекты, которыми
`network` управляет.

## Важно: тут появляются QObject и сигналы-слоты

Нижние слои (`models`, `interface`, `services`) — чистый C++ без Qt-механизмов.
**`core` — первый слой, где включается Qt-инфраструктура:** `ClientSession`
наследует `QObject` и общается через сигналы. Причина одна: он оборачивает
`QWebSocket`, а тот — асинхронный `QObject`, который шлёт сигналы о входящих
данных. Дальше по стеку (`network`) это тоже QObject-мир. Так что граница
«чистый C++ / Qt-событийная модель» проходит ровно здесь.

## Три класса

| Файл | Класс | Одной фразой |
|---|---|---|
| `ClientSession.*` | `ClientSession` | Одно WebSocket-соединение с клиентом |
| `ConferenceRoom.*` | `ConferenceRoom` (+ `ChatEntry`) | Один живой эфир: участники, рассылка, история чата |
| `RoomRegistry.*` | `RoomRegistry` | Реестр «код → комната»: создаёт и удаляет комнаты |

## `ClientSession` — одно соединение

Обёртка над `QWebSocket`. Хранит присвоенный сервером `id`, имя, привязанный
аккаунт и ссылку на текущую комнату. Один объект = один подключённый клиент.

### Зачем нужна обёртка, а не голый QWebSocket

Две причины:

1. **Добавить контекст к событию.** Голый `QWebSocket` шлёт сигнал «пришёл
   текст» — но *от кого*? `ClientSession` перехватывает сигналы сокета и
   переизлучает свои, **добавляя указатель на себя**:

   ```cpp
   // сокет:   textMessageReceived(text)              — кто прислал, неизвестно
   // сессия:  textReceived(ClientSession *self, text) — теперь сервер знает, от кого
   ```

   Поэтому `ConferenceServer::onText(session, text)` сразу знает отправителя.

2. **Хранить состояние соединения** рядом с сокетом: id, имя, аккаунт,
   mic/cam, текущая комната.

### Сигналы (наружу, к серверу)

| Сигнал | Когда |
|---|---|
| `textReceived(self, text)` | Пришло текстовое сообщение (JSON-команда) |
| `binaryReceived(self, data)` | Пришёл бинарный фрейм (медиа-кадр) |
| `disconnected(self)` | Сокет закрылся |

Сервер подключает их в `onNewConnection` и дальше работает только с
`ClientSession`, не касаясь `QWebSocket` напрямую.

### Ключевые поля

| Поле | Смысл |
|---|---|
| `id` | Числовой id в рамках эфира. У аккаунта постоянный (`2³¹ + userId`), у анонима — из счётчика сервера. |
| `name` | Имя, под которым виден в комнате. |
| `userId` | Аккаунт из куки WS-хендшейка; `-1` — аноним. |
| `login` / `accountName` / `avatarVer` | Данные профиля (для списков допуска алиасов, имени, аватарки). |
| `room` | Текущая комната; `nullptr` до `join`. |
| `micOn` / `camOn` | Состояние медиа — раздаётся другим, чтобы рисовать заглушку/mic-off. |

### Отправка и защита от перегруза

- `sendJson(obj)` / `sendBinary(data)` — послать этому клиенту.
- `close()` — вежливо закрыть сокет (сигнал `disconnected` придёт обычным
  путём).
- В конструкторе стоит лимит входящего сообщения **2 МиБ**: самое крупное
  легитимное — чат с картинкой (~600 КБ base64), потолок с запасом отсекает
  попытки завалить сервер гигантским фреймом.

## `ConferenceRoom` — один живой эфир

Список участников + история чата + умение рассылать всем сообщения. Медиа
**не декодирует** — только пересылает байты как есть (это SFU-релей).

### Два вида комнат — по `ownerId`

Один класс покрывает оба сценария, различие — в поле `ownerId`:

| `ownerId` | Что за комната | Как создаётся |
|---|---|---|
| `-1` | **Разовая конференция.** Случайный код, живёт пока есть люди. | `RoomRegistry::createRoom()` по запросу HTTP API |
| `≥ 0` | **Личная комната.** Код и правила из `PersonalRoom` (база). | `createPersonal()` при входе владельца |

`ownerId ≥ 0` — сигнал сетевому слою: «применяй правила из
[`PersonalRoomService`](../services/README.md) (пароль, присутствие владельца)».
Сам `core` этих правил не проверяет — он только помнит, чья это комната.

### Рассылка

```cpp
void broadcastJson(const QJsonObject &obj, ClientSession *except = nullptr);
void broadcastBinary(const QByteArray &data, ClientSession *except = nullptr);
```

Проходит по участникам и шлёт каждому. `except` пропускает одного — обычно
самого отправителя (медиа-кадр не нужно возвращать тому, кто его прислал).
Чат, наоборот, рассылается **всем включая отправителя** — чтобы у всех была
одна лента.

`participantsJson()` собирает список участников для нового вошедшего (id, имя,
mic/cam, и `user_id`+`avatar` для авторизованных — клиент строит URL аватарки
сам).

### История чата (`ChatEntry`) — и почему она здесь, а не в models

`ChatEntry` (одно сообщение: отправитель, текст, время, картинка) объявлен
**в `ConferenceRoom.h`, а не в [`../models`](../models/README.md)** — сознательно:
история живёт только в памяти комнаты и **в базу не сохраняется**. После
перезапуска сервера переписка эфира теряется. (Персистентный чат — задача на
будущее, тогда появится и модель, и таблица.)

Новому участнику история отдаётся в `join_ok` (`historyJson()`), чтобы он видел
разговор, шедший до него. Два ограничения памяти:

- **`kMaxHistory = 500`** — храним последние 500 сообщений, старые вытесняются.
- **`kMaxHistoryImages = 24`** — картинок держим только 24 свежих. У более
  старых `image` очищается, но остаётся пометка `imageDropped` — клиент
  покажет заглушку «картинка вытеснена». Текст сообщения при этом остаётся.
  Причина: картинки тяжелее текста на порядки, иначе история съела бы сотни
  мегабайт на комнату.

### Жизненный цикл: пустая / в эфире

Комната отслеживает два момента времени — от них зависит и уборка, и вид у
владельца на главной:

| Метка | Смысл | Когда 0 |
|---|---|---|
| `emptySinceMs` | С какого момента комната стоит пустой | пока внутри есть люди |
| `liveSinceMs` | С какого момента непрерывно не пуста | пока пусто |

`addParticipant` при первом входе ставит `liveSinceMs` и обнуляет
`emptySinceMs`; `removeParticipant` при выходе последнего — наоборот. По
`emptySinceMs` реестр решает, когда комнату убрать (см. ниже); `liveSinceMs`
показывает владельцу «в эфире · длительность».

## `RoomRegistry` — реестр и владелец комнат

Хеш `код → комната`. **Владеет** комнатами: создаёт (`new`) и удаляет
(`delete`).

| Метод | Что делает |
|---|---|
| `createRoom()` | Разовая комната со случайным уникальным кодом. |
| `createPersonal(code, ownerId)` | Личная комната с заданным кодом; `nullptr`, если код занят живой комнатой. |
| `find(code)` | Комната по коду или `nullptr`. |
| `purgeIdle(ttlMs)` | Удалить комнаты, простоявшие пустыми дольше `ttlMs`. |
| `roomCount()` | Сколько комнат сейчас. |

### Генерация кода (`generateCode`)

24 символа `[A-Za-z0-9]` ≈ **143 бита энтропии** от криптостойкого источника
ОС. Код нельзя подобрать перебором — поэтому ссылка-приглашение и есть «секрет»
комнаты, отдельный пароль разовой комнате не нужен. (Коды личных комнат/алиасов
короче — их генерирует [`PersonalRoomService`](../services/README.md) по другим
правилам.)

### Отложенная уборка — зачем пауза

Комната удаляется **не сразу** при выходе последнего участника, а через
`ttlMs` простоя (`purgeIdle` зовётся по таймеру из
[`../network`](../network/README.md)). Пауза даёт пережить обрыв связи
последнего участника без потери комнаты и истории чата: отвалившийся по сети
успевает вернуться в ту же комнату.

## Владение объектами — кто кого удаляет

Это самое важное для правильной работы с памятью. Три уровня владения:

```
ConferenceServer  ──владеет──►  ClientSession   (QObject-ребёнок сервера)
        │                            │
        │                   ClientSession ──владеет──► QWebSocket (setParent)
        │
RoomRegistry      ──владеет──►  ConferenceRoom
        │
ConferenceRoom    ──ХРАНИТ, но НЕ владеет──►  ClientSession (сырые указатели)
```

- **`ConferenceServer` владеет сессиями:** `ClientSession` создаётся как
  QObject-ребёнок сервера, сокет — ребёнок сессии. Разрушение каскадное.
- **`RoomRegistry` владеет комнатами:** создаёт и `delete`-ит их.
- **`ConferenceRoom` НЕ владеет участниками:** держит сырые `ClientSession*`
  только чтобы рассылать. Когда сессия отключается, `network` сам убирает её из
  комнаты (`removeParticipant`) до удаления объекта. **Комната никогда не
  удаляет сессии — иначе был бы двойной delete.**

Правило на будущее: если добавляешь ссылку на `ClientSession` куда-то ещё —
это **наблюдатель, не владелец**. Владелец один — сервер.

## Как этот слой используется

`core` пассивен: он не запускает событий сам, им управляет
[`../network/ConferenceServer`](../network/README.md). Типичный поток входа:

```
network: onNewConnection  → new ClientSession (обёртка сокета)
network: handleJoin       → registry.find(code) / createPersonal(...)
                          → room.addParticipant(session)
                          → room.broadcastJson({participant_joined})
network: onText(chat)     → room.appendChat(entry) + room.broadcastJson(...)
network: onDisconnected   → room.removeParticipant(session)
таймер:  purgeIdleRooms   → registry.purgeIdle(ttl)
```

Как именно `network` разбирает сообщения и какой протокол при этом
используется — в [`../network`](../network/README.md).
