# MeetUp — методичка для разработчика десктопного клиента (Qt6 / C++)

Документ описывает, как устроен и как работает с сервером текущий **web-клиент**
(`Server/web/`), и что нужно реализовать в десктопном клиенте на Qt6 C++, чтобы
он был полностью совместим: подключение, HTTP API, WebSocket-протокол,
бинарный медиапротокол, кодеки и сквозное шифрование (E2E).

Актуально на июль 2026 г. Исходники, по которым написан документ:

| Что | Где |
|---|---|
| Сервер: WebSocket-relay | `network/ConferenceServer.cpp` |
| Сервер: HTTP API (аккаунты, комнаты, аватарки, alias-ссылки) | `network/HttpApi.cpp`, `services/AuthService.cpp` |
| Сервер: личные комнаты и alias-ссылки | `services/PersonalRoomService.cpp` |
| Сервер: комнаты | `core/ConferenceRoom.cpp`, `core/RoomRegistry.cpp` |
| Сервер: хранилища (SQLite) | `interface/sqlite/`, точка сборки — `main.cpp` |
| Web-клиент: медиадвижок (кодеки, бинарный протокол, E2E) | `web/assets/meetup-media.js` |
| Web-клиент: подключение и API-обёртки | `web/assets/meetup-common.js` |
| Web-клиент: логика конференции | `web/conference.html` |

## 1. Архитектура в двух словах

Сервер — **релей (SFU без транскодирования)** на Qt6 (QWebSocketServer + свой
HTTP-сервер). Он:

- раздаёт web-клиент по HTTP и обслуживает JSON API (`/api/...`);
- принимает WebSocket-соединения и маршрутизирует сообщения по комнатам;
- **не декодирует и не разбирает медиа** — бинарные кадры пересылаются всем
  участникам комнаты «как есть», сервер лишь вставляет в заголовок ID отправителя;
- ничего не знает о ключах E2E — при включённом шифровании хранит и
  ретранслирует только шифротекст.

Все решения о кодеках, шифровании и отрисовке принимает **клиент**. Это значит,
что десктопный клиент обязан говорить на том же прикладном протоколе, что и
web-клиент, — иначе они друг друга не увидят.

```
клиент A ──ws──►┐
клиент B ──ws──►│  ConferenceServer ──► broadcast в комнату (кроме отправителя)
клиент C ──ws──►┘        │
                         └─ RoomRegistry (комнаты создаёт только HTTP API)
```

## 2. Подключение

### 2.1. Адреса и порты

| Сценарий | HTTP (API + статика) | WebSocket |
|---|---|---|
| Локальная разработка (без прокси) | `http://<host>:80` (или `--http-port`) | `ws://<host>:9000` (или `--ws-port`) |
| Прод за nginx (docker-compose) | `https://<host>/` | `wss://<host>/ws` |

Правило из web-клиента (`meetup-common.js`): если страница открыта по
`https`, WebSocket идёт **только** через `wss://<host>/ws` (TLS терминирует
nginx и проксирует на порт 9000); по `http` — напрямую `ws://<host>:9000`.

Для десктопного клиента: дайте пользователю настройку «адрес сервера», и из неё
стройте два URL — `https://host/api/...` + `wss://host/ws` (прод) или
`http://host:80` + `ws://host:9000` (локально). Сертификат в проде
**самоподписанный** — в `QNetworkAccessManager`/`QWebSocket` понадобится
обработчик `sslErrors` с осознанным `ignoreSslErrors()` (лучше — по отпечатку
сертификата, а не «глотать всё»).

### 2.2. Жизненный цикл входа в конференцию

1. **Создать или проверить комнату по HTTP** (WebSocket-join в несуществующую
   комнату вернёт ошибку — комнаты создаёт только HTTP API):
   - создать: `POST /api/rooms` → `{"room": "<код из 24 символов>"}`;
   - проверить существующую: `GET /api/rooms/<код>` → `200` либо `404`.
2. **Открыть WebSocket** и первым сообщением отправить `join` (JSON, текстовый
   фрейм).
3. Получить `join_ok` — в нём ваш `sender_id`, список участников и история чата.
4. Слать/принимать: JSON-фреймы (чат, состояние) и бинарные фреймы (медиа).

Код комнаты — 24 символа `[A-Za-z0-9]` (~143 бита энтропии): сама ссылка и есть
«секрет» комнаты. Пустая комната живёт **10 минут** и удаляется сборщиком;
обрыв связи и возврат участника в этот интервал комнату не убивают.

### 2.3. Переподключение (как делает web-клиент)

- До **8 попыток**, задержка `min(attempts, 4) × 1.5 c`.
- После реконнекта заново отправляется `join`. **Анониму** сервер выдаст
  **новый `sender_id`** (id со счётчика, между подключениями не сохраняется);
  **авторизованный** (кука в WS-хендшейке, см. 2.4) каждый раз получает
  **один и тот же** `sender_id`.
- После `join_ok`: сбросить всех накопленных пиров-декодеров, отправить
  `KEYFRAME_REQ` (бинарный тип 6), заново разослать своё состояние `state`,
  продолжить захват.

### 2.4. Авторизованное подключение (кука в WS-хендшейке)

Если в HTTP-хендшейке WebSocket пришёл заголовок `Cookie` с живой сессией
`meetup_session=<токен>` (см. 3.2), сервер привязывает соединение к аккаунту:

- `sender_id` **постоянный**: `0x80000000 + user_id`, одинаков во всех
  комнатах и между переподключениями (анонимные id живут в нижней половине
  диапазона quint32 — коллизий нет);
- имя участника берётся **из профиля**, поле `name` в `join` игнорируется
  (можно слать пустым);
- повторный `join` того же аккаунта в ту же комнату **вытесняет старое
  соединение**: ему приходит `{"type":"error","reason":"session_replaced"}`,
  затем сервер закрывает сокет. Получив `session_replaced`, **не
  переподключайтесь автоматически** — иначе два устройства будут выбивать
  друг друга по кругу. Остальные участники видят `participant_left` и следом
  `participant_joined` с тем же id.

В Qt: `QNetworkRequest req(url); req.setRawHeader("Cookie",
"meetup_session=<токен>"); webSocket->open(req);` — либо просто общий
`QNetworkCookieJar` (браузер шлёт куку в хендшейке сам).

## 3. HTTP API (JSON)

Все ответы — JSON. При ошибке тело `{"error": "<машинный_код>"}`; текст для
пользователя подбирает клиент. Content-Type запросов — `application/json`.

### 3.1. Аккаунты

| Метод и путь | Тело запроса | Успех | Ошибки (`error`) |
|---|---|---|---|
| `POST /api/auth/register` | `{"login", "password", "display_name"}` | `200` + `{"user": {...}}` + кука | `invalid_login`, `invalid_display_name`, `weak_password`, `login_taken` (409), `invalid_json` |
| `POST /api/auth/login` | `{"login", "password"}` | `200` + `{"user": {...}}` + кука | `wrong_credentials` (401), `invalid_json` |
| `POST /api/auth/logout` | — | `200`, кука стирается (`Max-Age=0`) | — |
| `GET /api/me` | — | `200` + `{"user": {...}}` | `no_session` (401) |
| `PATCH /api/me` | `{"display_name": "..."}` | `200` + `{"user": {...}}` | `no_session` (401), `invalid_display_name`, `nothing_to_update` |
| `POST /api/me/avatar` | `{"image": "<base64 JPEG>"}` | `200` + `{"user": {...}}` | `no_session` (401), `bad_image`, `storage_error` (500) |
| `DELETE /api/me/avatar` | — | `200` + `{"user": {...}}` | `no_session` (401) |
| `GET /api/users/<id>/avatar` | — | `200` + `image/jpeg` (тело — файл) | `no_avatar` (404) |

Объект `user` (публичное представление, без секретов):

```json
{ "id": 1, "login": "anna", "display_name": "Анна", "avatar_ver": 0 }
```

**Аватарки.** Файл один на пользователя (JPEG), перезаписывается на месте.
`avatar_ver` — счётчик: `0` — аватарки нет, растёт с каждой загрузкой. Картинку
берите по `GET /api/users/<id>/avatar`, добавляя версию в URL
(`?v=<avatar_ver>`) — раздача помечена «кэшировать вечно», и версия в URL даёт
свежий файл после замены. `POST` принимает JPEG в base64 (сервер проверяет
сигнатуру `FF D8 FF` и лимит размера). В конференции id и версия аватарки
приходят в `participant_joined` / `participants` (поля `user_id` и `avatar`,
см. 4.2).

Правила валидации (сервер, `AuthService`):

- `login`: нормализуется `trim + lowercase`; после — `^[a-z0-9_.-]{3,32}$`;
- `display_name`: 1–40 печатных символов;
- `password`: 8–128 символов.

Хранение пароля на сервере: PBKDF2-HMAC-SHA512, 64 000 итераций, соль 16 байт —
клиента это не касается (пароль уходит открытым текстом по HTTPS), но полезно
знать: **вход и регистрация занимают ~0.1–0.3 с** — это намеренно.

### 3.2. Сессия

- Выдаётся в куке **`meetup_session`**: `HttpOnly; SameSite=Lax; Path=/;
  Max-Age=2592000` (30 дней, скользящее продление при активности).
- Токен — 32 случайных байта в base64url.
- В Qt: `QNetworkAccessManager` с установленным `QNetworkCookieJar` подхватит и
  будет слать куку сам. Сервер **проверяет куку и в WebSocket-хендшейке** —
  авторизованное соединение получает постоянный id и имя из профиля (см. 2.4).
- Хранилище — **SQLite** (`mount/meetup.db`): аккаунты, сессии и личные комнаты
  переживают перезапуск сервера. Всё равно закладывайтесь на `no_session` в
  любой момент (сессия истекает через 30 дней, сервер её мог почистить, куку
  могли сбросить) — при нём предлагайте войти заново, а не падайте.

### 3.3. Комнаты

| Метод и путь | Успех | Ошибки |
|---|---|---|
| `POST /api/rooms` | `{"room": "<код>"}` | — |
| `GET /api/rooms/<код>` | см. ниже | `room_not_found` (404) |

`GET /api/rooms/<код>` для разовой конференции отвечает
`{"room": "<код>", "participants": N}`. Для **личной комнаты** (см. 3.4) —
даже когда она не в эфире:

```json
{ "room": "vi5", "personal": true, "title": "Комната Игоря",
  "has_password": true, "online": false, "participants": 0 }
```

`online: true` означает «владелец сейчас в комнате» — только тогда гостей
пускают. По `has_password` клиент понимает, что пароль нужно спросить **до**
`join`, а не после первой ошибки.

### 3.4. Личная комната владельца — `/api/me/room`

Постоянная комната пользователя: свой код в ссылке (`?room=vi5`), название и
необязательный пароль. Одна на аккаунт. Все маршруты требуют сессию
(`no_session`, 401).

| Метод и путь | Тело запроса | Успех | Ошибки (`error`) |
|---|---|---|---|
| `GET /api/me/room` | — | `200` + `{"room": {...}}` | `no_room` (404) |
| `POST /api/me/room` | `{"code", "title", "password"}` | `200` + `{"room": {...}}` | `invalid_code`, `code_taken` (409), `room_exists` (409), `invalid_title`, `invalid_password` |
| `PATCH /api/me/room` | любое подмножество `code/title/password` | `200` + `{"room": {...}}` | те же + `no_room` (404) |
| `DELETE /api/me/room` | — | `200` | `no_room` (404) |
| `POST /api/me/room/close` | — | `200` + `{"kicked": N}` | `no_room` (404) |

`POST /api/me/room/close` завершает конференцию: сервер шлёт всем участникам
`{"type":"error","reason":"room_closed"}` и закрывает их сокеты — комната сама
угаснет по TTL. `DELETE` удаляет комнату вовсе (участники внутри остаются, но
по коду больше никого не пускают).

Объект `room` в ответах владельцу (пароль виден — это осознанное решение,
владелец вправе его «посмотреть»; хранится открытым текстом, это пароль
комнаты, а не аккаунта):

```json
{ "code": "vi5", "title": "Комната Игоря", "password": "kot123",
  "online": false, "participants": 0,
  "participant_names": [], "live_since_ms": 0 }
```

`participant_names` и `live_since_ms` (когда начался текущий эфир) сервер
добавляет только владельцу — чтобы показать на главной, кто сейчас в комнате.

Валидация (`PersonalRoomService`): `code` нормализуется `trim + lowercase`,
после — `^[a-z0-9_-]{3,32}$`; `title` — 1–64 символа; `password` — до 128
(пустой = входа по паролю нет; `PATCH {"password": ""}` снимает пароль).

### 3.5. Alias-ссылки на личную комнату — `/api/me/room/aliases`

Дополнительные коды входа в **ту же** личную комнату, но со своими правилами:
отдельный пароль, лимит использований (одноразовая ссылка), список допущенных
логинов, вкл/выкл. До 5 на комнату; код генерирует сервер. Гость входит по
`?room=<код алиаса>` — сервер применяет правила алиаса вместо правил самой
комнаты. Все маршруты требуют сессию.

| Метод и путь | Тело запроса | Успех | Ошибки (`error`) |
|---|---|---|---|
| `GET /api/me/room/aliases` | — | `200` + `{"aliases": [ {...} ]}` | `no_session` (401) |
| `POST /api/me/room/aliases` | `{"password", "uses", "logins", "enabled"}` | `200` + `{"alias": {...}}` | `no_room` (404), `alias_limit` (409), `invalid_password`, `invalid_uses`, `invalid_logins` |
| `PATCH /api/me/room/aliases/<id>` | любое подмножество тех же полей | `200` + `{"alias": {...}}` | `no_alias` (404) + те же |
| `DELETE /api/me/room/aliases/<id>` | — | `200` | `no_alias` (404) |

Объект `alias` (владельцу — целиком, включая пароль):

```json
{ "id": 3, "code": "a1b2c3d4e5", "password": "kot123",
  "uses_left": -1, "logins": ["anna"], "enabled": true }
```

- `uses` — число (лимит входов) или `null`/`-1` = без лимита; в ответе поле
  зовётся `uses_left` и уменьшается с каждым состоявшимся входом гостя
  (дойдя до нуля, алиас удаляется). Реконнект того же аккаунта расход не
  тратит.
- `logins` — массив логинов; пустой = пускать любого, иначе только
  перечисленных (анонимов список отсекает всегда — у них нет логина).
- `enabled: false` — выключенный алиас неотличим от несуществующего.

Для десктоп-клиента это опциональная фича владельца; смотреть/проверять комнату
по коду алиаса можно тем же `GET /api/rooms/<код>` (см. 3.3) — вернётся
`personal: true` с паролем именно этого алиаса.

## 4. WebSocket: текстовый протокол (JSON)

Текстовые фреймы — JSON-объекты с полем `type`.

### 4.1. Клиент → сервер

```jsonc
{ "type": "join", "room": "<код комнаты>", "name": "<имя, ≤64>",
  "password": "<пароль личной комнаты>" }   // name игнорируется у авторизованных (см. 2.4);
                                            // password нужен только гостю личной комнаты с паролем
{ "type": "chat", "text": "<сообщение, ≤2000>",
  "image": "<base64 JPEG>" }               // image необязателен; можно text, image или оба
{ "type": "state", "mic": true, "cam": false }   // любое подмножество полей
{ "type": "screen", "on": true }           // закрепить/отпустить демонстрацию экрана (см. 4.3)
{ "type": "ping", "t": 1751970000000 }     // замер RTT: сервер вернёт t в pong как есть
```

После ошибки `join` (неверный пароль, комната не в эфире) сокет **остаётся
открытым** — можно отправить `join` повторно, не переподключаясь.

Картинка в чате — JPEG в base64 (web ужимает до ~480 тыс. символов); слишком
большая даёт ошибку `image_too_large`. Шифрование картинки — как у текста
(см. 7.3).

### 4.2. Сервер → клиент

```jsonc
// Только вошедшему, в ответ на join:
{ "type": "join_ok",
  "sender_id": 7,
  "participants": [ { "id": 3, "name": "Анна", "mic": true, "cam": true,
                      "user_id": 5, "avatar": 2 }, ... ],   // user_id/avatar — только у авторизованных
  "history":      [ { "sender_id": 3, "sender_name": "Анна",
                      "text": "привет", "timestamp_ms": 1751970000000,
                      "image": "<base64 JPEG>", "image_dropped": false }, ... ],  // image* — если было
  "room_title": "Комната Петра",   // только у личной комнаты; показать в шапке
  "screen_id": 3 }                 // только если в комнате уже идёт демонстрация экрана

// Остальным участникам:
{ "type": "participant_joined", "id": 7, "name": "Пётр", "mic": true, "cam": true,
  "user_id": 9, "avatar": 1 }      // user_id/avatar — только у авторизованных (avatar лишь при версии > 0)
{ "type": "participant_left",   "id": 7 }
{ "type": "participant_state",  "id": 7, "mic": false, "cam": true }

// Всем, включая отправителя (единая лента чата):
{ "type": "chat", "sender_id": 7, "sender_name": "Пётр",
  "text": "привет", "timestamp_ms": 1751970000000,
  "image": "<base64 JPEG>" }       // image — только если сообщение с картинкой

// Демонстрация экрана — всем, включая инициатора (см. 4.3):
{ "type": "screen", "id": 7, "on": true }

// Ответ на ping — только спросившему:
{ "type": "pong", "t": 1751970000000 }   // то же t, что в ping — RTT считает клиент

// Ошибки:
{ "type": "error", "reason": "invalid_json" | "unknown_type" | "invalid_join"
                            | "already_joined" | "room_not_found" | "not_in_room"
                            | "session_replaced"    // см. 2.4 — не реконнектиться!
                            | "room_closed"         // владелец завершил конференцию — не реконнектиться
                            | "wrong_password"      // личная комната: пароль не подошёл — спросить и повторить join
                            | "alias_forbidden"     // alias-ссылка со списком логинов: этот аккаунт не допущен
                            | "screen_busy"         // демонстрацию уже ведёт другой — отпустить свой экран
                            | "image_too_large"     // картинка чата больше лимита — показать уведомление
                            | "room_offline" }      // личная комната без владельца: ждать и повторять join (web — раз в 5 с)
```

Замечания:

- История чата — последние **500** сообщений комнаты (картинок из них хранится
  максимум 24 свежих; у вытесненных приходит `image_dropped: true` без самой
  картинки). Отдаётся один раз в `join_ok` — новый участник видит разговор до
  него.
- `room_not_found`, `session_replaced`, `room_closed`, `alias_forbidden` web
  трактует как фатальные (расходимся без реконнекта); `wrong_password` уводит
  на гейт-экран, `room_offline` — в ожидание; неизвестный `reason` — просто
  логировать и не падать (список ошибок расширяемый).
- На `participant_joined` вещающий клиент **принудительно шлёт keyframe**
  (иначе новичок ждал бы опорный кадр до 3 секунд).
- На `participant_state` с `cam:false` — убрать последний кадр этого участника
  (показать заглушку).

### 4.3. Демонстрация экрана

Одна демонстрация на комнату. Клиент просит закрепить её за собой
(`{"type":"screen","on":true}`) или отпускает (`{"on":false}`). Подтверждение —
широковещательное `{"type":"screen","id":<кто>,"on":<bool>}` всем, включая
инициатора; если экран занят другим — `error` `screen_busy`. Кадры экрана идут
отдельным бинарным потоком (типы `SCREEN_CODED`/`SCREEN_JPEG`, см. 5.2) и
рисуются в отдельную плитку, параллельно камере того же участника.

- `screen_id` в `join_ok` сообщает вошедшему, что демонстрация уже идёт (и от
  кого) — чтобы сразу завести под неё декодер и плитку.
- Сервер релеит кадры экрана **только** от закреплённого ведущего; кадры от
  остальных молча отбрасываются (могли не успеть получить отказ `screen_busy`).
- Ведущий отвалился (`participant_left`) — сервер сам шлёт `screen id on:false`,
  слот освобождается. После реконнекта посреди своей демонстрации повторите
  `{"type":"screen","on":true}` — прежний слот на сервере уже освободился.

## 5. WebSocket: бинарный протокол v2 (медиа)

Все числа — **little-endian**.

### 5.1. Формат кадра

Клиент **отправляет** (заголовок 11 байт):

```
[type:1][flags:1][codec:1][timestamp:8 LE][payload...]
```

Клиент **получает** (заголовок 15 байт — сервер вставил `sender` после первого
байта и больше ничего не трогал):

```
[type:1][sender:4 LE][flags:1][codec:1][timestamp:8 LE][payload...]
```

### 5.2. Константы

```cpp
// Типы сообщений (type)
enum : quint8 {
    MSG_VIDEO_JPEG   = 1,   // legacy: JPEG-кадр камеры
    MSG_AUDIO_PCM    = 2,   // legacy: PCM 16 кГц Int16 моно
    MSG_VIDEO_CODED  = 3,   // кадр видеокодека камеры (payload = EncodedVideoChunk)
    MSG_AUDIO_CODED  = 4,   // пакет аудиокодека (payload = Opus-пакет)
    MSG_SCREEN_CODED = 5,   // кадр видеокодека демонстрации экрана (см. 4.3)
    MSG_KEYFRAME_REQ = 6,   // просьба всем вещающим прислать опорный кадр (payload пуст)
    MSG_SCREEN_JPEG  = 7,   // legacy: JPEG-кадр демонстрации экрана
};

// Флаги (flags), битовая маска
enum : quint8 {
    FLAG_KEYFRAME  = 1,     // это опорный кадр (для VIDEO_CODED и SCREEN_CODED)
    FLAG_ENCRYPTED = 2,     // payload зашифрован E2E (см. раздел 7)
};

// Идентификаторы кодеков (codec)
enum : quint8 {
    CODEC_VP8  = 1,         // "vp8"
    CODEC_H264 = 2,         // "avc1.42E01F" — H.264 Baseline L3.1, поток Annex B!
    CODEC_VP9  = 3,         // "vp09.00.10.08" — VP9 profile 0
    CODEC_OPUS = 4,         // "opus"
};
```

### 5.3. Семантика полей

- `timestamp` (uint64 LE) — **`Date.now()` в миллисекундах** (стенные часы
  отправителя) для **всех** типов кадров: и кодек-чанков (видео/экран/аудио),
  и legacy (JPEG/PCM). Одна шкала у аудио и видео — не случайность: приёмник по
  ней **синхронизирует губы** (придерживает видеокадр, обогнавший звук из
  джиттер-буфера, см. 6.4). Поэтому:
  - **не** кладите сюда PTS энкодера в микросекундах (`performance.now()`,
    счётчик сэмплов) — это разъедет картинку и звук у web-получателя. PTS для
    декодера ведите отдельной внутренней шкалой, наружу отправляйте `Date.now()`;
  - метки **разных участников** между собой сравнивать всё равно нельзя (часы у
    всех свои) — картинка выравнивается только с собственным звуком того же
    отправителя;
  - приёмник дополнительно использует метку как PTS для декодера — монотонности
    достаточно, `Date.now()` её обеспечивает.
- `codec` заполняется для `VIDEO_CODED`/`SCREEN_CODED`/`AUDIO_CODED`; для
  legacy-типов и `KEYFRAME_REQ` — `0`.
- `KEYFRAME_REQ` шлётся с пустым payload и **ретранслируется всем** участникам
  комнаты; получив его, вещающий клиент помечает следующий кадр как keyframe
  (у web-клиента — с защитой от спама: не чаще 1 раза в 500 мс на отправку,
  сам запрос — не чаще 1 раза в секунду).
- Сервер проверяет только «фрейм ≥ 9 байт» и первый байт не разбирает контент:
  всё после `type` для него непрозрачно.

### 5.4. Приём: правила устойчивости (так делает web-клиент)

- **Один декодер на участника** (`sender`) и на тип потока. Если пришёл кадр с
  другим `codec` — пересоздать видеодекодер.
- После (пере)создания видеодекодера ждать кадр с `FLAG_KEYFRAME`; дельта-кадры
  до него молча выбрасывать и отправить `KEYFRAME_REQ`.
- Ошибка декодера → закрыть, пересоздать, снова ждать keyframe + `KEYFRAME_REQ`.
- Участник ушёл (`participant_left`) → освободить его декодеры/плееры.
- Если от участника > 5 с не было кадров — показать заглушку вместо
  «замороженного» кадра.

### 5.5. Отправка: параметры и backpressure

Видео (web-клиент, стоит повторить в Qt):

| Параметр | Камера | Демонстрация экрана |
|---|---|---|
| Максимум | 1280×720 | нативное |
| Битрейт | 1 200 000 бит/с | 1 500 000 бит/с |
| FPS | 24 | 15 |
| Keyframe | каждые 72 кадра (~3 с) + по `KEYFRAME_REQ` | так же |
| Размеры | **чётные** (`w & ~1`, `h & ~1`) | так же |
| Режим | минимальная задержка (`latencyMode: realtime`, без B-кадров) | так же |

Битрейт и FPS в таблице — пресет по умолчанию («med»). У web три пресета
качества, переключаемых на лету (энкодер пересоздаётся со следующим кадром):

| Пресет | Камера (битрейт / fps) | Экран (битрейт / fps) | Аудио (битрейт) |
|---|---|---|---|
| low  | 400 000 / 15  | 600 000 / 5    | 16 кбит/с |
| med  | 1 200 000 / 24 | 1 500 000 / 15 | 32 кбит/с |
| high | 2 500 000 / 30 | 4 000 000 / 30 | 64 кбит/с |

Разрешение камеры при этом задаёт захват, а не пресет. Десктоп-клиенту
достаточно одного разумного набора (med); пресеты — приятная опция, не
требование протокола.

Аудио: Opus, 48 000 Гц, **моно**, 32 кбит/с (пресет med), кадры
**20 мс (960 сэмплов)**.

Backpressure (обязательно, иначе на слабой сети всё развалится):

- если в сокете скопилось **> 1.5 МБ** неотправленного (`bufferedAmount` в
  браузере; в Qt — считайте через `QWebSocket::bytesWritten` разницу
  отправлено/подтверждено) — **пропускать видеокадры** (аудио продолжать);
- если очередь энкодера растёт (в WebCodecs `encodeQueueSize > 2` для видео,
  `> 4` для аудио) — пропускать кадр, а не копить.

### 5.6. Legacy-формат (JPEG/PCM)

Фолбэк web-клиента для браузеров без WebCodecs: JPEG-кадры (payload — целый
JPEG-файл) и PCM 16 кГц Int16 LE моно (`MSG_AUDIO_PCM`). Камера — `MSG_VIDEO_JPEG`
≤480×360 ~10 к/с; демонстрация экрана — `MSG_SCREEN_JPEG` ≤1280×720 ~2 к/с
(крупнее и реже, иначе текст на экране нечитаем). Десктопному клиенту
**отправлять** legacy не нужно (кодеки всегда доступны), но **принимать**
JPEG/PCM желательно уметь — вдруг в комнате старый браузер. Минимум: JPEG
декодируется `QImage::fromData`, PCM играется напрямую.

## 6. Кодеки: что выбрать в Qt

### 6.1. Матрица совместимости

Web-клиент пробует кодеки в порядке **H.264 → VP8 → VP9** и берёт первый
поддержанный (H.264 почти везде аппаратный). Десктопный клиент должен:

- **декодировать все три** видеокодека + Opus (принимать можно от кого угодно);
- **кодировать хотя бы один** видеокодек из списка + Opus.

Важно про H.264: web-клиент настроен на `avc: { format: "annexb" }` — то есть
поток **Annex B** (стартовые коды `00 00 00 01`, SPS/PPS внутри keyframe, без
отдельного `avcC` extradata). Это удачно: FFmpeg-декодер H.264 ест Annex B
нативно, и энкодеру нужно выдавать то же самое (для x264/openh264 это режим по
умолчанию при raw-потоке; главное — включить `repeat-headers`/`global_header`
**выключен**, чтобы SPS/PPS повторялись в каждом keyframe).

Профиль `avc1.42E01F` = Baseline, level 3.1 — без B-кадров (это и нужно для
задержки).

### 6.2. Рекомендуемые библиотеки

| Задача | Рекомендация | Альтернатива | Комментарий |
|---|---|---|---|
| Захват камеры | **Qt Multimedia** (`QCamera` + `QMediaCaptureSession` + `QVideoSink`) | libwebcam/FFmpeg avdevice | `QVideoSink::videoFrameChanged` отдаёт `QVideoFrame`; `map()` → YUV/NV12 планы для энкодера |
| Захват микрофона | **Qt Multimedia** (`QAudioSource`, формат 48000 Гц / mono / Int16 или Float) | PortAudio, miniaudio | Просите сразу 48 кГц — не будет ресемплинга |
| Захват экрана | **Qt Multimedia** (`QScreenCapture`, Qt ≥ 6.5) | платформенные API | тоже уходит в `QVideoSink` |
| Видео H.264 | **openh264** (Cisco, BSD) или FFmpeg/libavcodec | libx264 (**GPL!**) | openh264 — самый простой лицензионно; FFmpeg удобен тем, что закроет и декод всех форматов |
| Видео VP8/VP9 | **libvpx** (BSD) | FFmpeg (внутри тот же libvpx) | если берёте FFmpeg — отдельный libvpx не нужен |
| Декодирование видео (все кодеки) | **FFmpeg / libavcodec** | libvpx + openh264 по отдельности | один API на всё: H.264/VP8/VP9, плюс `sws_scale` для конвертации в RGB |
| Аудио Opus | **libopus** (BSD) | FFmpeg | эталонная реализация; API из 4 функций (`opus_encoder_create/encode`, `opus_decoder_create/decode`) |
| Ресемплинг (если вдруг не 48к) | **libswresample** (FFmpeg) | libsamplerate, speexdsp | |
| Вывод звука | **QAudioSink** + свой джиттер-буфер | PortAudio | см. 6.4 |
| Отрисовка видео | `QVideoFrame` → `QVideoSink` (QML `VideoOutput`) или `QImage` + `QPainter` | OpenGL-виджет | для начала хватит `QImage` RGB через `sws_scale` |
| JSON | **QJsonDocument** (в Qt) | nlohmann/json | |
| WebSocket | **Qt WebSockets** (`QWebSocket`) | libwebsockets | `binaryMessageReceived` / `textMessageReceived` — ровно наша модель |
| HTTP | **QNetworkAccessManager** | libcurl | кука сессии — через `QNetworkCookieJar` |
| Крипто (E2E) | **OpenSSL 3 (libcrypto, EVP API)** | Botan | нужен именно PBKDF2-SHA256 + AES-256-GCM; в libsodium **нет PBKDF2** — для совместимости с web он не подходит |

Итоговый минимальный набор зависимостей: **Qt6 (Core, Network, WebSockets,
Multimedia) + FFmpeg (avcodec, avutil, swscale, swresample) + libopus +
OpenSSL**. FFmpeg при сборке должен включать декодеры `h264`, `vp8`, `vp9` и,
если кодировать через него, энкодер (`libopenh264` или `libvpx`).

Лицензии: libvpx/openh264/libopus/FFmpeg(LGPL-сборка без x264) совместимы с
закрытым клиентом при динамической линковке; **libx264 — GPL**, его в
проприетарный клиент не берите.

### 6.3. Конвейер отправки (видео)

```
QCamera → QVideoSink → QVideoFrame (map, NV12/I420)
   → (throttle до 24 fps, чётные размеры, пропуск при backpressure)
   → энкодер (H.264 Annex B / VP8), keyframe каждые 72 кадра или по запросу
   → [E2E seal, если включено]
   → пакет v2: [3][flags][codec][ts_мс:8][payload] → QWebSocket::sendBinaryMessage
     (ts = Date.now() в мс — общая шкала с аудио, см. 5.3)
```

### 6.4. Конвейер приёма (аудио) и джиттер-буфер

Web-клиент собирает приём так: очередь декодированных чанков по 960 сэмплов,
**предбуфер 3 чанка (~60 мс)** перед стартом воспроизведения; если очередь
выросла больше 12 чанков (~240 мс лага) — срезать до 6. В Qt то же самое
делается поверх `QAudioSink` в pull-режиме или через свой `QIODevice`.
Микшировать участников можно простым сложением float-сэмплов с клампом.

Индикатор «говорит»: RMS по кадру > 0.02 (по нормализованным float-сэмплам) —
подсветить участника на ~450 мс.

## 7. Сквозное шифрование (E2E)

Сервер не участвует: ключ живёт только у клиентов, включение — опция комнаты
«по договорённости» (каждый участник вводит одну и ту же фразу или входит по
ссылке с ключом). Участник без ключа продолжает получать трафик, но не может
его расшифровать (web-клиент показывает плитку «🔒 Зашифровано — нужен ключ»).

### 7.1. Ключ

Два способа получить один и тот же 256-битный ключ AES:

1. **Из фразы** (все участники вводят одинаковую):

   ```
   key = PBKDF2-HMAC-SHA256(
       password = UTF-8(фраза),
       salt     = UTF-8("meetup-e2e-v1|" + roomCode),
       iterations = 150000,
       dkLen    = 32 байта)
   ```

   Соль включает код комнаты — одна и та же фраза в разных комнатах даёт
   разные ключи.

2. **Из ссылки-приглашения**: raw 32 байта ключа в **base64url без паддинга**
   в URL-fragment: `conference.html?room=<код>#k=<base64url(key)>`. Fragment
   не отправляется на сервер браузером — на это и расчёт. Десктопный клиент
   должен уметь распарсить такую ссылку (и генерировать её для «поделиться»).

В OpenSSL: `PKCS5_PBKDF2_HMAC(phrase, len, salt, saltLen, 150000, EVP_sha256(), 32, key)`.

### 7.2. Шифрование медиа-payload

Алгоритм: **AES-256-GCM**. Шифруется **только payload** бинарного кадра;
заголовок `[type][flags][codec][ts]` остаётся открытым (серверу нужен `type`,
приёмнику — `codec` и флаг keyframe до расшифровки). В `flags` ставится
`FLAG_ENCRYPTED` (2).

Формат зашифрованного payload:

```
[IV:12][ciphertext...][GCM-tag:16]      // tag приклеен к концу шифротекста
```

- **IV (12 байт)** = `[префикс:4][счётчик:8 LE]`, где префикс — 4 случайных
  байта на сессию отправки (генерируются при установке ключа), счётчик
  инкрементируется на каждый пакет. Это гарантирует уникальность IV в пределах
  ключа — для GCM это критично.
- **AAD (2 байта)** = `[type, codec]` исходного сообщения. Приёмник при
  расшифровке передаёт те же два байта — подмена типа/кодека по пути ломает
  аутентификацию.
- Расшифровка неудачна (не тот ключ / битые данные) — пакет молча
  выбрасывается; после **3 неудач подряд** от одного участника web-клиент
  помечает его «заблокирован ключом» (и снимает пометку при первом успешном
  пакете, счётчик сбрасывается).
- При установке/смене ключа: сбросить счётчики неудач всех пиров и
  **принудительно отправить keyframe** (старые дельта-кадры бесполезны).

В OpenSSL это `EVP_aes_256_gcm()`:
`EVP_EncryptInit_ex` → `EVP_CIPHER_CTX_ctrl(EVP_CTRL_GCM_SET_IVLEN, 12)` →
задать IV → `EVP_EncryptUpdate(nullptr, ..., aad, 2)` (AAD) →
`EVP_EncryptUpdate(payload)` → `EVP_EncryptFinal_ex` →
`EVP_CIPHER_CTX_ctrl(EVP_CTRL_GCM_GET_TAG, 16, tag)`, tag дописать в конец.

### 7.3. Шифрование чата

Чат ходит **текстовым** JSON (`type: "chat"`), поэтому шифротекст упаковывается
в строку:

```
"🔒e2e:" + base64url( IV(12) || ciphertext || tag(16) )
```

- Префикс — **байты UTF-8 строки `🔒e2e:`** (`F0 9F 94 92 65 32 65 3A` — 8
  байт). В C++ сравнивайте по байтам UTF-8, не по «символам».
- Параметры AES-GCM те же, что для медиа, но AAD = `[250, 0]` (250 —
  зарезервированный «тип» для чата, codec 0).
- Сервер хранит такую строку в истории как обычный текст — новый участник с
  ключом расшифрует историю из `join_ok`; без ключа показывается заглушка
  «Зашифрованное сообщение — нужен ключ». Web-клиент хранит для каждого
  сообщения исходную строку (`raw`), чтобы **перечитать чат, когда ключ введут
  позже** — сделайте так же.
- Строка без префикса — обычное открытое сообщение (шифрование опционально и
  посреди разговора может включаться/выключаться).
- **Картинка чата** (`image`) шифруется так же — тот же формат
  `"🔒e2e:" + base64url(...)`, но AAD = `[251, 0]` (251 — «тип» для картинки).
  Открытая картинка едет обычным base64 JPEG (клиент сам добавляет
  `data:image/jpeg;base64,`). Как и текст, храните `raw` картинки, чтобы
  перечитать её после ввода ключа.

### 7.4. Порядок операций — важно

Шифрование асинхронным быть не обязано (в C++ оно синхронное — проще), но
web-клиент вынужден сериализовать отправку в «цепочки» на канал (аудио/видео
отдельно), чтобы пакеты не перемешивались. В Qt просто шифруйте и отправляйте
в одном потоке в порядке кодирования.

При входе в комнату web-клиент **сначала** восстанавливает ключ (из `#k=` или
сохранённого секрета) и только **потом** подключается — иначе история чата из
`join_ok` распакуется без ключа. Повторите этот порядок.

## 8. Аккаунты и личные комнаты: что уже реализовано

Всё описанное в разделе 3 работает и живёт в SQLite. Коротко, как это
складывается в сценарии, и что стоит поддержать десктоп-клиенту.

- **Аккаунты** — регистрация, вход, `GET/PATCH /api/me`, аватарки, сессия в
  куке (3.1–3.2). Web: `login.html` при живой сессии сразу уводит в лобби;
  `anon_lobby.html` работает и без аккаунта — анонимный вход остаётся полноценным
  сценарием, аккаунт не обязателен.
- **Личные комнаты** (3.4) — постоянный код-ссылка на аккаунт, название,
  необязательный пароль. Комнату «открывает» вход владельца; гостей пускают,
  только пока владелец в эфире (иначе `room_offline`). Пароль проверяется при
  `join` полем `password`, отказ — `wrong_password` (спросить и повторить `join`
  по тому же сокету, см. 4.1). Владелец видит живое состояние комнаты и может
  завершить её (`/api/me/room/close`).
- **Alias-ссылки** (3.5) — приглашения в ту же комнату со своими правилами:
  свой пароль, лимит использований (одноразовость), белый список логинов
  (`alias_forbidden` для недопущенных). Опциональная фича — десктоп-клиенту
  достаточно уметь входить по такому коду как по обычному.

Советы на будущее (совместимость вперёд):

- Не завязывайтесь жёстко на набор полей `join` — сервер может добавить новые;
  неизвестные поля в JSON игнорируются.
- Держите обработку `error.reason` расширяемой: незнакомый код — показать как
  текст/лог и не падать (список в 4.2 будет пополняться).

Идея на будущее (пока **не** спецификация): выводить ключ E2E из пароля комнаты
той же PBKDF2-схемой — тогда пароль давал бы и допуск, и шифрование. Сейчас
допуск (пароль комнаты) и E2E (фраза/ссылка `#k=`) независимы.

## 9. Скелет клиента на Qt6: примеры кода

### 9.1. Упаковка/распаковка бинарного кадра

```cpp
#include <QByteArray>
#include <QtEndian>

QByteArray packV2(quint8 type, quint8 flags, quint8 codec,
                  quint64 tsMs, const QByteArray &payload)   // tsMs — Date.now() в мс, см. 5.3
{
    QByteArray out;
    out.reserve(11 + payload.size());
    out.append(char(type));
    out.append(char(flags));
    out.append(char(codec));
    quint64 tsLe = qToLittleEndian(tsMs);
    out.append(reinterpret_cast<const char *>(&tsLe), 8);
    out.append(payload);
    return out;
}

struct FrameV2 {
    quint8 type = 0, flags = 0, codec = 0;
    quint32 sender = 0;
    quint64 ts = 0;
    QByteArray payload;
};

bool unpackV2(const QByteArray &d, FrameV2 &f)   // фрейм, пришедший от сервера
{
    if (d.size() < 15) return false;
    f.type   = quint8(d[0]);
    f.sender = qFromLittleEndian<quint32>(d.constData() + 1);
    f.flags  = quint8(d[5]);
    f.codec  = quint8(d[6]);
    f.ts     = qFromLittleEndian<quint64>(d.constData() + 7);
    f.payload = d.mid(15);
    return true;
}
```

### 9.2. E2E: ключ и AES-GCM (OpenSSL 3)

```cpp
#include <openssl/evp.h>
#include <openssl/rand.h>

// Ключ из фразы — совместимо с web-клиентом.
QByteArray deriveKey(const QString &phrase, const QString &roomCode)
{
    const QByteArray pass = phrase.toUtf8();
    const QByteArray salt = ("meetup-e2e-v1|" + roomCode).toUtf8();
    QByteArray key(32, 0);
    PKCS5_PBKDF2_HMAC(pass.constData(), pass.size(),
                      reinterpret_cast<const unsigned char *>(salt.constData()), salt.size(),
                      150000, EVP_sha256(),
                      key.size(), reinterpret_cast<unsigned char *>(key.data()));
    return key;
}

// Ключ из ссылки #k=... (base64url без '=')
QByteArray keyFromUrlFragment(const QString &b64url)
{
    return QByteArray::fromBase64(b64url.toLatin1(),
                                  QByteArray::Base64UrlEncoding);
}

class E2eCipher {
public:
    explicit E2eCipher(QByteArray key) : m_key(std::move(key)) {
        RAND_bytes(m_ivPrefix, 4);
    }

    // payload -> [iv:12][ct][tag:16]; aad = {type, codec}
    QByteArray seal(quint8 type, quint8 codec, const QByteArray &plain) {
        unsigned char iv[12];
        memcpy(iv, m_ivPrefix, 4);
        quint64 ctrLe = qToLittleEndian(m_counter++);
        memcpy(iv + 4, &ctrLe, 8);

        QByteArray out(12 + plain.size() + 16, 0);
        memcpy(out.data(), iv, 12);

        const unsigned char aad[2] = { type, codec };
        EVP_CIPHER_CTX *ctx = EVP_CIPHER_CTX_new();
        int len = 0;
        EVP_EncryptInit_ex(ctx, EVP_aes_256_gcm(), nullptr,
                           reinterpret_cast<const unsigned char *>(m_key.constData()), iv);
        EVP_EncryptUpdate(ctx, nullptr, &len, aad, 2);
        EVP_EncryptUpdate(ctx, reinterpret_cast<unsigned char *>(out.data()) + 12, &len,
                          reinterpret_cast<const unsigned char *>(plain.constData()), plain.size());
        int fin = 0;
        EVP_EncryptFinal_ex(ctx, reinterpret_cast<unsigned char *>(out.data()) + 12 + len, &fin);
        EVP_CIPHER_CTX_ctrl(ctx, EVP_CTRL_GCM_GET_TAG, 16,
                            out.data() + 12 + plain.size());
        EVP_CIPHER_CTX_free(ctx);
        return out;
    }

    // Возвращает пустой QByteArray при неверном ключе/битых данных.
    QByteArray open(quint8 type, quint8 codec, const QByteArray &sealed) {
        if (sealed.size() < 12 + 16) return {};
        const int ctLen = sealed.size() - 12 - 16;
        QByteArray out(ctLen, 0);
        const unsigned char aad[2] = { type, codec };

        EVP_CIPHER_CTX *ctx = EVP_CIPHER_CTX_new();
        int len = 0;
        EVP_DecryptInit_ex(ctx, EVP_aes_256_gcm(), nullptr,
                           reinterpret_cast<const unsigned char *>(m_key.constData()),
                           reinterpret_cast<const unsigned char *>(sealed.constData()));
        EVP_DecryptUpdate(ctx, nullptr, &len, aad, 2);
        EVP_DecryptUpdate(ctx, reinterpret_cast<unsigned char *>(out.data()), &len,
                          reinterpret_cast<const unsigned char *>(sealed.constData()) + 12, ctLen);
        EVP_CIPHER_CTX_ctrl(ctx, EVP_CTRL_GCM_SET_TAG, 16,
                            const_cast<char *>(sealed.constData()) + 12 + ctLen);
        int fin = 0;
        const bool ok = EVP_DecryptFinal_ex(ctx,
                          reinterpret_cast<unsigned char *>(out.data()) + len, &fin) == 1;
        EVP_CIPHER_CTX_free(ctx);
        return ok ? out : QByteArray();
    }

private:
    QByteArray m_key;
    unsigned char m_ivPrefix[4];
    quint64 m_counter = 0;
};
```

Чат: `"🔒e2e:" + QString::fromLatin1(sealed.toBase64(QByteArray::Base64UrlEncoding | QByteArray::OmitTrailingEquals))`,
где `sealed = cipher.seal(250, 0, text.toUtf8())`. Картинка чата — так же, но
`sealed = cipher.seal(251, 0, jpegBytes)` (AAD-тип 251, см. 7.3).

## 10. Рекомендуемый порядок реализации

1. **HTTP + комнаты**: создать/проверить комнату, вход по коду (curl уже
   позволяет проверить сервер: `curl -X POST http://host/api/rooms`).
2. **WebSocket + JSON**: join, список участников, чат (без шифрования). Уже на
   этом шаге можно тестировать против web-клиента в соседней вкладке.
3. **Аудио**: захват 48к → libopus 20 мс → пакеты v2 → приём/декодирование/
   джиттер-буфер. Аудио проще видео и сразу даёт «живую» связь.
4. **Видео-приём**: декодирование H.264/VP8/VP9 (FFmpeg) + логика
   keyframe/KEYFRAME_REQ; отрисовка. Отправлять пока нечего — web-клиент
   вещает, десктоп смотрит.
5. **Видео-отправка**: захват камеры → энкодер → пакеты. Проверка: web-клиент
   видит десктоп.
6. **E2E**: ключ из фразы/ссылки, seal/open медиа и чата (+ картинок), состояние
   «заблокировано» для пиров с чужим ключом.
7. **Аккаунты**: login/register-экраны, кука в `QNetworkCookieJar`, аватарки,
   личные комнаты и вход в них по паролю.
8. **Демонстрация экрана**: захват (`QScreenCapture`) → второй видео-поток
   (типы `SCREEN_*`), протокол `screen`/`screen_busy`/`screen_id` (см. 4.3),
   отдельная плитка на приёме.
9. Полировка: переподключение, смена устройств, картинки в чате, `ping`-индикатор,
   индикаторы «говорит».

## 11. Чек-лист подводных камней

- [ ] Всё числовое в бинарном протоколе — **little-endian**.
- [ ] Отправляете заголовок **11 байт**, принимаете **15** (sender вставляет сервер).
- [ ] `timestamp` в кадрах — **`Date.now()` в мс** (не мкс!), одна шкала у аудио и видео — иначе поедет синхронизация губ.
- [ ] `sender_id` меняется при каждом join/реконнекте у анонима — не кэшировать между сессиями (у авторизованного он постоянный).
- [ ] H.264 — только **Annex B**, SPS/PPS в каждом keyframe, Baseline, без B-кадров.
- [ ] Размеры видеокадра — чётные.
- [ ] После пересоздания декодера — ждать keyframe, слать `KEYFRAME_REQ` (с rate-limit 1/с).
- [ ] На `participant_joined` — принудительный keyframe со своей стороны (rate-limit 500 мс).
- [ ] Демонстрация экрана — отдельные типы `SCREEN_CODED`/`SCREEN_JPEG` и протокол `screen` (см. 4.3); свои декодер и плитка.
- [ ] E2E: IV уникален (префикс+счётчик), AAD `[type, codec]`, tag в конце, заголовок не шифруется.
- [ ] PBKDF2-**SHA256**, 150 000 итераций, соль `"meetup-e2e-v1|"+roomCode` — ровно так, иначе ключи не совпадут с web.
- [ ] base64url **без паддинга** для `#k=` и чата; картинка чата — AAD-тип 251, текст — 250.
- [ ] Префикс чата `🔒e2e:` сравнивать по байтам UTF-8.
- [ ] Ключ восстанавливать **до** обработки `join_ok` (иначе история чата не расшифруется).
- [ ] Backpressure: >1.5 МБ в сокете — пропуск видеокадров.
- [ ] Комнаты создаются только по HTTP; пустая комната умирает через 10 минут. Исключение — личная комната: её открывает join владельца.
- [ ] Данные в SQLite (переживают рестарт), но `no_session` всё равно возможен (истечение/чистка) — предлагать войти заново, не падать.
- [ ] Самоподписанный сертификат в проде — валидация по отпечатку, а не отключение проверки.

## 12. Как проверить совместимость с web-клиентом

1. Поднять сервер (Docker: `docker compose up -d --build`; локально: `run.ps1`).
2. В браузере открыть `https://<IP>/`, войти без аккаунта, создать комнату,
   скопировать ссылку (при включённом E2E — «Ссылка с ключом E2E»).
3. Десктопным клиентом войти в ту же комнату (код — из URL `?room=...`, ключ —
   из `#k=...`).
4. Проверить в обе стороны: видео, звук, демонстрацию экрана, чат (текст и
   картинки), историю чата, вход без ключа (должны быть «заблокированные»
   плитки), смену ключа на лету. Для аккаунтов — регистрация/вход, аватарка,
   личная комната с паролем.
